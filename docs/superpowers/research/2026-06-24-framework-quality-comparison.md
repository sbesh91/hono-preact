# Framework code-quality & modularity comparison (rerun)
_2026-06-24: rerun of the 2026-06-18 comparison at hono-preact v0.8_

This is a rerun of the 2026-06-18 internal assessment, re-judged against current HEAD (hono-preact v0.8, the #139-#176 wave) rather than v0.7. The competitor checkouts are unchanged: react-router 8.0.1 (`4cf6e62`), `@sveltejs/kit` 2.67.0 (`81f253e`), solid-start 2.0.0-alpha.3 (`47d5e6a`). The discipline is the same: judge only the comparable concern in each subsystem, like-for-like; do not credit hono-preact for features the peers lack (RSC, single-fetch, SPA mode, more adapters) nor penalize peers for complexity that serves real features; treat SvelteKit's JS+JSDoc runtime as qualitative on type safety, not a raw cast count; and treat a gap as closed only on positive evidence in current source or a run test. Symbol/file renames since June (`ActionStub`->`ActionRef`, `RouteServer`->`RouteBinder`, and the extracted `stream-pump.ts` / `document-shell.ts` / `compose-server-chain.ts` / `node-request.ts` / `outcome-translation.ts`) have been mapped onto the current tree. Six subsystems are covered: the original five plus the net-new Realtime subsystem.

## Verdict

The honest bottom line has moved in hono-preact's favor since June, and the move is real rather than cosmetic. The original report's verdict was "the internals are not poor quality and are not poorly modularized; hono-preact wins or ties the large majority of lenses, with weaknesses that are specific and concentrated, not systemic." At HEAD that holds more strongly: the two named soundness gaps are fixed, the headline type-rigor deficits are closed, and the localized duplication the original flagged as the recurring weakness has largely been factored into single engines. No subsystem regressed.

What closed. The routing optional-param matcher divergence, the original report's only **high**-severity correctness gap (a `/files/:id?` route not matching `/files` server-side, an edge-case guard bypass), is gone: the matcher now iterates `len = max(ps, us)` and handles each modifier, the exact regression is pinned by a passing test, and `grep` confirms the old `ps.length === us.length` line is nowhere in `packages/server/src`. The typed-routes engine, which had zero type-level tests (the clearest maturity gap versus react-router and SvelteKit), now has a 151-line tsc-enforced `typed-routes.test-d.ts` plus an app-level companion; it ran clean ("no errors"). The `RouteParams` hyphenated-name unsoundness is fixed by a `ParamNameChar`/`IsParamName` grammar that provably mirrors the runtime regex. On the data side, the one lens a peer genuinely beat hono-preact on (react-router's `Serialize<T>` wire-mirror type) is closed: `Serialize<T>` now models the JSON round-trip at every client boundary and is pinned by a 167-line type-level suite that uniquely probes that the consumer return types actually apply it, a boundary-wiring discipline react-router's own tests do not all carry. The decoded-outcome 7-kind switch and `validateTimeoutMs` duplication are both eliminated by real extraction (one `applyDecodedOutcome` dispatcher with verified exactly-two callers; one `timeout.ts` source). In SSR, PR #138 extracted the streaming pump and document shell out of `renderPage`, which is now a 195-line orchestrator whose streaming branch is a single delegating call, with the ~10 repeated abort guards collapsed to one `guardedWrite`. In middleware, the original's two losses shared one root cause (`dispatchServer`/`dispatchClient` duplication plus per-handler chain reimplementation); both are gone, replaced by one `dispatch<Ctx,T>` engine and one `compose-server-chain.ts` composition home, and the auth-bypass footgun is now fail-closed-at-construction on the loader and action handlers. In build, the riskiest dev path (Node-req to Fetch translation) was extracted to pure `node-request.ts` functions with 7 unit cases, and the `importKind` double-casts are gone.

What is newly strong. The net-new **Realtime** subsystem (WebSocket sockets, typed channels, presence rooms, topic pub/sub, and a Cloudflare Durable Object backend) is, on its own merits, among the better-engineered parts of the framework: `room-engine.ts` is a transport-agnostic protocol engine that both the Node runtime and the DO drive through one `RoomTransport` interface, so the room protocol cannot drift between runtimes; `ws-lifecycle.ts` is the same single-engine win on the client; the workerd boundary is clean (`realtime-do.ts` is the sole file with a runtime `cloudflare:workers` import, verified, so all platform-free CF logic is unit-testable in plain vitest); type-level coverage is strong (373 lines of `.test-d.ts` with 17 negative assertions); and the core unit layer ran 95 tests green in well under a second. No peer ships a comparable first-party realtime subsystem (SvelteKit's nearest analog is one-directional query SSE, not duplex/presence/pub-sub), so this is a feature the peers lack, judged on the framework's own bar.

What remains weak, and it must be said plainly because the concern is whether the internals are quietly poor. They are not, but three things are genuinely unresolved. First, a real contract asymmetry: `socketsHandler.resolvePageUse` is still optional with a silent `?? (() => [])` fallback, reintroducing the exact dropped-page-guard footgun that the loader and action handlers now fail-closed on, on a sibling handler built on the same engine, and no test pins that branch. This is the single most important remaining issue, because it is the same auth-bypass class the report has flagged twice. Second, persistent god-file pressure in two specific files: `define-routes.tsx` is still a 616-line file with six tree-walkers (two of them, `flattenTree` and `buildInnerRoutes`, structurally parallel where one engine would do; line count did not drop), and `sockets-handler.ts` is a 494-line file owning registry build, collision assert, resolution, guard probe, and a 3-way connector dispatch with six `socketDef!` non-null assertions. Third, cross-cutting mutable global state is real and concentrated: the data layer keeps a process-global shared-cache registry (with a candid multi-tenant caveat) plus a module-global action-result store, and the realtime subsystem keeps five `Symbol.for`/module-level singletons; each is a deliberate, documented HMR/injection seam with a test reset, not incidental state, but it is real global state. Smaller residuals: `<Form>` still cannot drive a streaming action and the limitation is undocumented in source; the whole build codegen layer is untyped string concatenation (caught only downstream, never by tsc); the deny-path action payload stays untyped; and several Cloudflare-only failure modes (the 6KB forward-header budget) are invisible in Node development.

Net: the scorecard improved from the June baseline. Routing moved from comparable-with-real-soundness-edges to comparable-with-those-edges-closed; the data layer's one losing lens (type safety) moved to parity; SSR holds ours-better and is now better-decomposed; middleware moved from comparable (2-2 against react-router) to ours-better as both losing lenses flipped; build holds ours-better; and Realtime enters strong. Confidence is high on the closures (each was re-verified by reading current source and, where fast, running the specific test) and appropriately tempered on the "ours-better" verdicts by the same scope caveat as June: several wins are partly a function of a smaller surface area that has not yet forced the scaling failure modes (god-files, cross-build state) the mature peers carry.

### Net-new public surface since 2026-06-18

Two substantial additions landed after the June report, both verified at HEAD.

**Standard Schema validation across the data layer.** `validate.ts` (`validateWithSchema`/`normalizeIssues`/`mapIssuesToFields`) is the shared runner. `defineAction`'s `{ input: schema }` overload infers `InferOutput` as the payload type; loaders gain `paramsSchema`/`searchSchema` that narrow the route-derived param type (`define-loader.ts`); `<Form schema>` does client pre-validation. The server enforces it as `deny(422)` + `VALIDATION_ISSUES_KEY` at the chain's innermost layer (`page-actions-handler.ts`), and `getValidationIssues` + `<FieldError>` surface it. One asymmetry to flag (detailed under 5.2): that issue-surfacing path is action-only; a loader's `paramsSchema`/`searchSchema` failure throws `deny(404)`/`deny(400)` and discards the issues (`loader-schema.ts:36-45`), so the loader path returns a bare 400/404 with no `VALIDATION_ISSUES_KEY`, while `<Form schema>` client pre-validation does surface field issues. Type-level coverage: `define-action-input.test-d.ts`, `define-loader-schema.test-d.ts`, `form-schema.test-d.ts`; a real server e2e (`standard-schema-validation.test.ts`) drives the action handler with Zod through 422 + issues + coercion and ran green.

**The entire Realtime subsystem** (PRs #139/#145/#150/#157/#166/#168/#170), net-new with no prior verdict to reconcile:
- _Typed duplex sockets:_ `defineSocket()` + `useSocket()` with full Incoming/Outgoing/Data inference and a per-connection `ServerSocket` handle (`define-socket.ts`, `use-socket.ts`; pinned by `define-socket.test-d.ts`).
- _Typed channels:_ `defineChannel('name/:param')<Payload>()` producing a branded `Topic<Payload>` over the same `/:param` grammar as routes (`define-channel.ts`).
- _Presence rooms:_ `defineRoom(channel, handler)` + `useRoom()` with server-mediated broadcast, a reactive presence roster (members/self), and channel-derived `ctx.params` (`define-room.ts`, `use-room.ts`, `room-engine.ts`).
- _Topic pub/sub:_ `publish()` + a swappable `PubSubBackend` (in-process default) feeding live loaders via `subscribeTopic`'s coalescing async generator.
- _Cloudflare Durable Object backend:_ a hibernating `HonoPreactRealtimeDO` multiplexing room/socket/topic/publish, an edge forward connector with header-smuggle defense (set/delete `x-hp-kind`) and a 6KB header budget, AsyncLocalStorage-scoped DO-backed pub/sub, and disjoint `room:`/`topic:` DO id prefixes. Denied/key-failed handshakes close at the edge (code 4403) without ever contacting the DO (verified at `realtime-do-glue.ts`).

This is a feature class the peers lack: a grep for `WebSocketPair`/`acceptWebSocket`/`DurableObject`/`defineRoom`/presence-runtime across all three competitor checkouts returns only incidental textual matches (SvelteKit's closest surface is one-directional query SSE in `remote.js`, not a duplex/presence/pub-sub subsystem).

## By subsystem

### 5.1 Routing (route definition, matching, manifest, typed params)

**Verdict: comparable (high confidence, verified).** Original 2026-06-18 was comparable across all four lenses; that holds at HEAD. The two original headline gaps are genuinely closed by verified evidence: the optional-param matcher divergence (a real server-side guard bypass) and the absence of any type-level tests on the typed-params engine.

**Gaps that remain:**

- `packages/iso/src/define-routes.tsx` is still a 616-line god-file (line count unchanged since the original report) carrying six tree-walkers: `collectRouteViolations` (:184), `collectServerImports` (:223), `collectServerRoutes` (:237), `collectRouteUse` (:277), `buildInnerRoutes` (:481), `flattenTree` (:529). `buildInnerRoutes` and `flattenTree` remain two structurally-parallel walkers differing mainly in emission target (VNode vs FlatRoute) and the layout-group `/*`-pair. Shared `joinRoutePath` (:130) and `composeUse` (:139) helpers and the `ROUTE_RULES` predicate table (:155) reduced but did not eliminate the duplication. **Fix:** unify the two walkers behind one generic tree-fold parameterized by an emit callback (`(node, joinedPath) => T`), so the view-vs-layout-vs-bare-grouping branching and the `/*`-pair emission live once.
- The `buildInnerRoutes` walker (the inner-router VNode emitter feeding `makeLayoutGroupComponent`) has no direct unit test; it is exercised only at render/integration level. Note: the verifier corrected an over-claim here, `flattenTree`'s layout-group / `/*`-pair / `/`-reset / bare-grouping branch emission IS directly unit-tested at `packages/iso/src/__tests__/define-routes.test.tsx:373-484`. So the accurate remaining gap is narrower than the raw judge claimed: one of the two parallel walkers is directly tested, the other only indirectly. **Fix:** add a focused unit test enumerating `buildInnerRoutes` emission for nested layout groups, mirroring the existing `flattenTree` cases.
- `interpolatePattern` (`packages/iso/src/internal/interpolate-pattern.ts:18-24`) silently truncates an absent required param for loose callers; safety depends on the strict `buildPath` overload plus a rooms-handler re-check rather than the primitive itself. **Fix:** add a `{ strict?: true }` runner variant that throws on a missing required segment, and have the loose call sites opt into it where they can.
- `findBestPattern` / `patternScore` are O(patterns) linear scans per server-side RPC resolution (`packages/server/src/route-pattern.ts:80-82`, documented). Fine at app scale, un-memoized. **Fix:** only if a generated route table grows large; a first-segment bucket index would cut the scan.

**Genuine strengths:** The server matcher is one pure, well-documented engine (`route-pattern.ts`, three functions: `urlPathMatchesPattern`/`patternScore`/`findBestPattern`) mirroring preact-iso's exec grammar exactly and shared by every server resolver (page-use, page-actions, byModuleKey), so client and server agree on matching. The exact guard-bypass regression is pinned and green (`route-pattern.test.ts`, 19 cases including `/files/:id?` resolving for `/files`; verified run). The typed-routes engine now has genuine, tsc-enforced type-level coverage (`typed-routes.test-d.ts`, 151 lines, verified "Type Errors no errors") plus an app-level companion against the real registered tree, which closes the original headline maturity deficit and brings the subsystem to par with React Router and SvelteKit. The type grammar provably mirrors the runtime regex (`ParamNameChar`/`IsParamName` demote `:foo-bar`/`:a.b` to literals contributing no param), so the type cannot over-claim a param the runtime ignores. Cast discipline is real: every residual `as` is a sanctioned, commented boundary (user-module export reads, preact-iso loose generics, `exec`'s `any` return reshaped via a typed `execParams` helper), and `interpolatePattern` uses a `seg is string` predicate.

*Competitor footnote (verification-supported):* React Router's matching machinery lives in a 2347-line `utils.ts` bundling flatten/rank/explode/compile/match/generatePath, larger than hono-preact's god-file; its param engine is equally mature (colocated `Expect<Equal<>>` tests, `ParamNameChar` including `-` agreeing with its runtime `[\w-]` regex). SvelteKit keeps its matcher in one tight 309-line `routing.js` with a more exhaustive unit suite, but that tracks larger feature scope (named matchers, unicode escapes) and its typed params are codegen `$types.d.ts` that stay `string` (no value narrowing). SolidStart trails clearly: `{ path: string } & Record<string, any>`, `toPath()` throws "Not implemented", `@ts-ignore` at the matching seam.

### 5.2 Data layer (loaders, actions, forms, caching, optimistic, validation)

**Verdict: ours-better (high confidence, verified; raw was ours-better).** The original verdict was ours-better at medium confidence with one lens (type safety) scored ours-worse because React Router shipped `Serialize<T>` and hono-preact did not. That single deficit is now closed, lifting type safety to comparable-with-RR and raising overall confidence to high.

**Gaps that remain:**

- `<Form>` still cannot drive a streaming action: `FormActionInput` pins `ActionRef<TPayload, TResult, never>` (`packages/iso/src/form.tsx:52`, verified) and the submit fetch sends `Accept: application/json` (:270); the server returns 405 for a streaming action hit without `text/event-stream` (`page-actions-handler.ts:243-248`). The `never` pin is now an honest type-level guarantee (a streaming ref is structurally inadmissible) rather than a silent runtime surprise, but there is still **no comment** near form.tsx:52/270 flagging the limitation (verified: grep for "stream" in form.tsx returns nothing). **Fix:** add a one-line comment at the `FormActionInput` definition and the fetch header explaining streaming actions are JSON-only here and why; consider a `@ts-expect-error`-style doc example.
- The deny-path payload stays untyped: `ActionResult` deny `data?: unknown` (`use-action-result.ts:18`/:67) and `StoredActionResult` `data: unknown` (`action-result-store.ts`), with several `source.x as TPayload` reads in `useActionResult` (`use-action-result.ts:57-73`). `getValidationIssues` guards the validation-issues subset soundly with a real predicate, but app-level deny data is unsound past that extractor. **Fix:** parameterize the stored deny payload on the action's declared error type (`defineAction` can thread a `TError`), or expose deny data only through a typed accessor that forces a user-supplied guard.
- Standard Schema validation surfaces issues on the action path only. A loader's `paramsSchema`/`searchSchema` failure throws `deny(404, 'Invalid route parameters')` / `deny(400, 'Invalid search parameters')` and discards the Standard Schema issues (`packages/iso/src/internal/loader-schema.ts:36-45`, verified), whereas the action path returns `deny(422)` + `VALIDATION_ISSUES_KEY`. ## Cross-cutting themes

**Recurring strengths that hold up under verification.** The most consistent and verified pattern is *one engine, many drivers*: the server route matcher (`route-pattern.ts`), the outcome dispatcher (`applyDecodedOutcome`), the middleware chain (`dispatch<Ctx,T>` + `compose-server-chain.ts`), the streaming wire producer/consumer (`stream-pump.ts` ↔ `stream-registry.ts` under one `StreamEvent` contract), the room protocol (`room-engine.ts` behind `RoomTransport`), and the client WS transport (`ws-lifecycle.ts`) are each single-sourced and shared across every consumer that would otherwise drift. This is the structural decision that most distinguishes hono-preact from the peers, whose comparable logic concentrates in god-file orchestrators (RR `router.ts` 7703 lines, RR `plugin.ts` 3869, SK `index.js` 1694, SK `client.js` 3449, all verified) carrying cross-cutting mutable state. The second recurring strength is *adversarial-unit-over-the-riskiest-code* test posture, verified by running the suites in every subsystem: the dangerous seams (matcher guard bypass, streaming backpressure/abort/XSS, the middleware called-once guard, the dual-key build parity, realtime reconnect/presence/fan-out) each have a fast targeted test, with heavy workerd/dev-server integration split out of the unit pool so it does not starve parallelism. The third is *cast discipline backed by real predicates*: across data, middleware, SSR, and realtime, the residual `as` casts are confined to sanctioned boundaries (untrusted JSON, symbol-keyed/hibernation stores, user-module exports, third-party loose generics), and value-narrowing is done with `value is T` predicates (`isAsyncGenerator`, `isValidationIssue`, `isServerImport`, `isTopicSubscriber`, the `runs==='server'` filter, `resolveRoomKey`'s per-value validation) rather than inline casts.

**Recurring weaknesses that survive the v0.8 cleanup.** Three classes persist. (1) *Density concentrates in a handful of branchy files that resisted extraction:* `define-routes.tsx` (616 LOC, two parallel tree-walkers), `action.ts`'s `useAction` mutate callback (lines 253-519), `use-loader-runner.tsx`'s reader-construction conditional (221-334), and `sockets-handler.ts` (494 LOC). None is a god-file by peer standards, but each is the local hotspot and each is a candidate for the same extract-and-share treatment the cleanup already applied elsewhere. (2) *Untyped string-concat codegen persists in the build layer* (`generateCoreAppModule`, the stub templates, both `wrapEntry` bodies): mitigated by imported contract constants and structural test assertions, but the type checker still cannot validate an emitted module. Notably this is peer-shared (RR codegens identically), so it is a genuine weakness rather than a competitive gap. (3) *Type-level test coverage is uneven:* the data and typed-routing and realtime subsystems have strong `*.test-d.ts` suites with negatives, but the middleware chain (ordering, `ServerCtx<S>`) and the SSR/streaming public types (`StreamEvent`, `ViewRenderArgs`) have none, so a regression that loosened those types would pass `test:types`. A fourth, narrower survivor worth flagging because it is the only one that touches correctness rather than polish: the *socket-resolver contract asymmetry* (`socketsHandler.resolvePageUse` optional with a silent `?? (() => [])` fallback, untested) reintroduces the exact dropped-page-guard auth-bypass class the loader and action handlers were hardened against, on a sibling handler built on the same engine. It is safe in the generated entry but is the one remaining place where a cross-cutting concern (auth/permission middleware) can be silently dropped, and it has no test pinning the omitted-resolver branch.

## Recommendations

Priority-ordered, highest-impact first. Every item below is verified still-open against the gap ledger; nothing already shipped is recommended.

1. **Close the socket-resolver auth-bypass asymmetry (correctness, P0-class).** Make `resolvePageUse` required and construction-validated on `socketsHandler` exactly as on the loader and action handlers, and add a fail-closed-at-construction test plus a does-not-drop-a-guard test for the socket-upgrade path (`sockets-handler.ts:111-113,301`). This is the only remaining place a permission middleware layer can be silently dropped; it is safe today only because the generated entry happens to wire it.

2. **Add the missing type-level tests where a loosening regression would pass CI.** A `compose-server-chain.test-d.ts` pinning `[app,page,unit]` ordering and `ServerCtx<S>` narrowing, and a `stream-registry.test-d.ts` pinning the `StreamEvent` union and `ViewRenderArgs`. These are the two subsystems whose boundary types currently rest on runtime tests alone.

## Methodology & caveats

**Pipeline.** This rerun used a multi-agent profile → judge → verify → synthesize pipeline per subsystem. A profiling pass read the current source on disk and mapped each original 2026-06-18 gap onto the renamed/extracted HEAD files (e.g. `ActionStub`→`ActionRef`, `RouteServer`→`RouteBinder`, and the net-new `stream-pump.ts`/`document-shell.ts`/`compose-server-chain.ts`/`node-request.ts`/`outcome-translation.ts`). A judging pass assigned per-lens verdicts with paired ours/theirs citations. An independent verification pass then re-read the cited lines and, where fast, *ran* the specific tests rather than trusting the profile: it confirmed gap closures (matcher guard-bypass regression green, typed-routes `.test-d.ts` "no errors", the streaming and realtime unit suites green, zero-cast grep results) and corrected over-claims. Where the verifier's adjusted verdict differs from the raw judge verdict, this report follows the verified verdict; claims a verifier marked unsupported or imprecise were down-weighted or dropped. The most consequential correction was *in hono-preact's favor*: the raw judge claimed the `define-routes.tsx` walker branching had no direct unit test, but verification found `flattenTree`'s branch emission is directly tested (`define-routes.test.tsx:373-484`), so the routing test gap is narrower than first stated (only `buildInnerRoutes` is integration-only). Competitor-side line counts the judge cited loosely were corrected upward on verification (RR `utils.ts` is 2347 not ~1100, RR `router.ts` is 7703 not ~6900), which if anything strengthens the pro-hono-preact modularity contrast. No pro-hono-preact claim collapsed on reading competitor source; the verifiers' `grassIsGreener` checks came back empty or pointed the other way.

**Caveats.**

- *Maturity and scope normalization.* React Router, SvelteKit, and SolidStart are far more mature and carry features hono-preact lacks (RSC, single-fetch, SPA mode, many adapters). Each lens judged only the *comparable* concern in each subsystem; hono-preact was not credited for features the peers lack, nor were peers penalized for complexity that serves real features they ship. A competitor god-file is only cited as a weakness for the specific comparable concern that lives inside it.

- *SvelteKit is JS + JSDoc, not TS.* Its runtime type-safety is qualitative, assessed by reading its JSDoc annotations and `@ts-expect-error` suppressions, not by a raw cast count, and it is never penalized merely for being JS.

- *Competitors are at current main and structurally ~unchanged since June.* The peer god-files that anchor the modularity contrast are stable: RR `router.ts` 7703 lines and `plugin.ts` 3869 lines, SK `client.js` 3449 lines and `index.js` 1694 lines (all verified at HEAD of the local checkouts).

- *SolidStart is the exact same commit the original profiled* (47d5e6a, verified), so its assessment is a re-read of the same code, not a fresh release. React Router (8.0.1) and SvelteKit (2.67.0) are at their current pinned versions in `/tmp/fw-compare`.

- *Realtime has no like-for-like peer.* No peer ships a comparable first-party WebSocket/rooms/presence/pub-sub primitive, so its verdict rests on intrinsic merit against the framework's own bar, with the explicit understanding that the absence of a peer is not itself a quality result.
