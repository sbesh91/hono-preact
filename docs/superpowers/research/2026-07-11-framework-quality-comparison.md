# Framework code-quality & modularity comparison (second rerun)
_2026-07-11: second rerun, at hono-preact v0.10.1 (HEAD `60df3c21`). Baselines: the original 2026-06-18 report at v0.7 and the 2026-06-24 rerun at v0.8._

This is the second rerun of the internal assessment, re-judged at current HEAD after two major waves (the v0.9/v0.10 loader-inversion release train and the first-load asset-delivery cluster) plus a deliberate execution of the 2026-06-24 report's own backlog. Unlike the first rerun, the competitor checkouts are NOT the June ones: react-router 8.2.0 (`ac5d9d5`), `@sveltejs/kit` 2.69.2 (`5c38e51`), and `@solidjs/start` 2.0.0-alpha.3 (`ec1b82b`, same alpha, newer commit) were freshly cloned, and every peer figure cited below (line counts, mutable-state counts, test inventories) was re-measured at these versions rather than reused. The discipline is unchanged: judge only the comparable concern in each subsystem, like-for-like; no credit for features the peers lack and no penalty for complexity that serves real features (RSC, single-fetch, SPA mode, more adapters); SvelteKit's JS+JSDoc runtime is judged qualitatively on type safety; and a gap counts as closed only on positive evidence in current source or an executed test. Naming drift since June has been mapped onto the current tree (`sockets-handler.ts` split into `socket-resolution.ts` + a dispatch half; new `create-server-entry.ts`, `route-binding-guard.ts`, `loader-state.ts`, `server-route.ts`, `internal/route-change.ts`, and the vite asset files). Seven subsystems are covered: the six from June plus the net-new asset-delivery subsystem (#250/#252/#254/#266). Every verdict below is the verification-adjusted verdict: a separate adversarial pass re-read the cited lines, re-executed the cited suites directly (hundreds of tests across all seven subsystems, all green), and re-measured every competitor claim; judge claims the verifier marked unsupported or imprecise are dropped or narrowed here.

## Verdict

The bottom line moved again in hono-preact's favor, and for the first time every subsystem, including routing (comparable in both prior reports), verifies **ours-better** overall. Two things distinguish this rerun from June's. First, the June report's recommendations were executed rather than acknowledged: of the 29 gaps that report named, 15 verify closed, 8 partial, 5 open (one of those accepted by design), and 1 regressed, with the closures concentrated exactly where June said the risk was (the P0-class socket-auth item, loader validation parity, the sockets god-file, the missing type-level suites, the untested streaming pump). Second, honesty cut both ways at the lens level. Re-measuring the peers at their current versions corrected two of our own June-era over-claims and cost us cells: react-router's middleware unit suite (3,993 LOC, 81 adversarial tests including double-next and bubbling matrices) is a genuine match for ours, so middleware complexity/tests drops from ours-better to comparable; SvelteKit's published, typed `Builder` adapter interface is decisively stronger than our stringly `wrapEntry(): string` contract, so build API design drops to comparable. And the only **ours-worse** cell in this rerun, the first since the June rerun (the original 2026-06-18 scorecard carried four, all since closed), appears in the net-new asset subsystem: SvelteKit's asset-delivery API surface (per-asset preload filter, `preloadStrategy`, `inlineStyleThreshold`, full `base` support) beats ours, which currently breaks silently under a configured Vite `base`.

What closed, led by the item that mattered most. The June report's single P0-class finding, `socketsHandler.resolvePageUse` optional with a silent `?? (() => [])` fallback (the dropped-page-guard auth-bypass class, flagged twice across two reports), is closed and was verified by execution, not by PR title: the option is now a required field with an explicit contract (`socket-resolution.ts:117-126`), all three handlers validate it at construction through one shared `asserts`-predicate (`page-use-guard.ts:19-30`) and throw before returning a handler, the fallback survives only inside test helpers, and the fail-closed construction tests ran green in this audit (`sockets-handler.test.ts:480-499`). Beyond it: loader `paramsSchema`/`searchSchema` failures now carry Standard Schema issues end-to-end into a public `LoaderValidationError` read by the same `getValidationIssues` actions use, with executed parity assertions on both the 400 and 404 loader paths; the 494-LOC sockets god-file was split exactly as prescribed (resolution vs dispatch, six `socketDef!` bangs replaced by a discriminated union, zero non-null assertions in either half); the cross-runtime realtime traps became compiler-enforced contracts (deep `ReadonlyData`, factory-less `conn.data` unified to `undefined` down to workerd e2e, 23 `@ts-expect-error` negatives); the routing walker duplication was folded into one generic `walkRouteTree`; `streamDocumentResponse` gained a direct adversarial harness proving backpressure by bounding generator pulls; and the missing type-level suites landed across the board (`compose-server-chain.test-d.ts`, `stream-registry.test-d.ts`, `loader-state.test-d.ts`, all executed at 0 type errors).

What is newly strong. The v0.10 loader inversion is the best new API work in the codebase: one `LoaderCtx` surface whose standalone vs route-bound shape is selected by an unexported unique-symbol marker default, so binding intent falls out of which constructor you call and authors never write a generic; wrong consumption form is a compile error via the `Live` discriminant with runtime defense for JS callers. The binding model's auth story is defense-in-depth done right: fail-closed boot guards for misbound `__routeId` (exact, wildcard-subtree, and registry variants, `route-binding-guard.ts:118-212`), `byPattern`-only guard resolution with the URL-fuzzy `byPath` mode deleted outright, request-time fail-closed composition (`composeServerChainOrFailClosed`), and attacker-model tests (guards resolved from the declared pattern, never the attacker-chosen POST URL; the registry fallback cannot rescue a route-gated action). The public server surface collapsed to a safe minimum: the three handlers left `@hono-preact/server`'s export map entirely, and the only construction story is the typed, unit-tested `createServerEntry` factory, which also structurally de-risks the codegen (the emitted core-app is now a ten-line shim over a typed factory instead of string-borne composition logic). The net-new asset subsystem's failure-policy design is the best in the repo: a three-tier taxonomy (degrade-with-warn for optimizations, loud warn plus non-memoized retry for render-critical CSS, a typed build-failing conservation error for the CSS splitter) enforced by type and tested per tier, and the Lightning CSS splitter's known sharp edges are pinned by a bisected-trigger regression test and a real-serializer round-trip equivalence test, not folklore comments. Test posture overall remains the framework's decisive edge: fast adversarial unit suites sit directly on the riskiest new code (the cold-navigation coordinator's three abandonment paths, malformed SSE frames, x-hp-* identity-spoof defense on both DO branches, memo-poisoning retry semantics), where SvelteKit still has zero unit specs on its 3,451-LOC client nav machine and its 758-LOC `render.js`, and SolidStart's only server-fn spec imports a module path that does not exist.

What is newly weak, stated plainly, because most of it lives in the new code. First, the sharpest finding of this rerun: `serverRoute(r).socket` and `.room` silently discard their route argument (`server-route.ts:201-202`). Binding for sockets/rooms comes only from module mount, so a `serverRoute('/admin').socket(...)` in a `src/server` registry module resolves to the RPC path and runs with no page-tier guards, and the boot guard's container list covers only `serverLoaders`/`serverActions`, so neither a boot error nor a dev warning fires. Two of the binder's four arms promise a route-binding semantic the runtime does not deliver, on the auth-relevant path; this is the direct successor to June's P0 in kind, though narrower in reach. Second, growth concentrated risk where extraction was supposed to retire it: `useAction`'s mutate callback regrew to ~300 of `action.ts`'s 696 lines (June: 267 of 522) even after the prescribed SSE-decode extraction landed, because the #221 schema gate and snapshot plumbing were added inside it; `use-loader-runner.tsx` (521 LOC) still performs subscription side effects during render; `route-change.ts` concentrates 13 module-level mutable bindings around a document-global scheduler (global-by-necessity and heavily tested, but the subsystem's peak local-reasoning burden); and `server-entry.ts` regressed outright, growing from three mixed concerns to four (349 -> 486 LOC), the one June recommendation not executed. Third, contract asymmetries inside the new surfaces: the loader RPC deny envelope drops the typed `DenyCode` the action envelope carries; `mutate`'s failure arm is a plain `Error` so deny status/code/issues never reach the imperative caller without pairing `useActionResult`; `ActionResult`'s deny `data?: unknown` is still untyped (June gap, only the cast hygiene improved); and the asset layer's cross-package artifact contract is comment-synced twin types plus byte-copied path helpers with no parity test. Fourth, unguarded edges: a configured Vite `base` silently turns route CSS into render-blocking 404s with no config-time guard (the analogous `cssCodeSplit` misconfiguration does get a hard error); `eventStream`'s payload queue is unbounded where the client hooks cap at 128 and warn; the dev-vs-prod route-CSS divergence (prod render-blocking, dev FOUC) is silent; and `expandGlobFs`, the only fs-touching discovery path for content-route hints, has zero direct tests.

Net: the scorecard improved again and its texture changed. Routing moved from comparable to ours-better (the fold, the boot guard, the type-level suites, and re-measured peer cores: react-router's `createRouter` is a single ~3,000-line closure holding 43 mutable bindings inside a 7,722-LOC file; SvelteKit's `client.js` is 3,451 LOC with 23 module-level lets and no unit spec). Data type safety moved from parity to ours-better (predicate-guarded wire decode and five executed data-layer type-level suites, of 18 package-wide in iso, against react-router's asserted turbo-stream decode), while data complexity/tests honestly settles at comparable against react-router's 33,413-LOC in-memory router test corpus. SSR holds ours-better on all four lenses, with one of our June-era claims corrected: react-router does ship type-level tests (about 50 inline `Expect<Equal>` assertions, 21 of them pinning its single-fetch data-channel serialization), so our SSR type-safety edge stands on the narrower, verified ground that RR does not type-pin its streaming wire internals while we pin the wire union itself. Middleware and build hold ours-better with the two cell concessions above. Realtime holds intrinsically strong and now has a partial peer to measure against (SvelteKit's experimental, default-off, one-way `query.live`), and it wins that overlap. Confidence is high across all seven subsystems; assets, the newest code, was raised to high by the verification pass despite its one lost lens. The June scope caveat, that several wins reflect a smaller surface that has not yet forced scaling failure modes, is weakening on the evidence: the surface grew substantially this cycle (710/696/669/586/583-LOC files) and the single-engine discipline held under that growth, but the three largest new files are also exactly where this report's new weaknesses concentrate, so the caveat is fading, not gone.

The CSS splitter's correctness posture is defense-in-depth: a safety invariant that only widens toward global on doubt, build-failing conservation asserts, a regression pin for the bisected Lightning CSS `var()`-serialization corruption with the trigger documented in the test, and a round-trip rule-set-equivalence test through the real minifier; the layout-evidence coincidental-substring hole was found, fixed, and tested as a before/after pair. Pure-over-injected-dependencies architecture makes the whole build-time layer unit-testable without a real Vite build: 135 tests across 12 files, re-run green in under 400ms per suite during this audit. Cast discipline is near-total for a subsystem parsing Babel ASTs, Lightning CSS ASTs, and untrusted JSON: one stray cast plus one sanctioned, runtime-filtered JSON boundary in ~2,300 LOC, with predicates everywhere else. And the Lightning CSS visitor quirks are recorded as probe-verified facts (flat RuleExit only; callbacks must return undefined) rather than folklore, exactly what the next maintainer upgrading lightningcss needs.

*Competitor footnote (verification-supported):* SvelteKit wins API design on this subsystem: a per-asset user preload filter (`resolve_opts.preload({type,path})`), a `preloadStrategy` escape hatch, `inlineStyleThreshold` CSS inlining, full base/relative-assets href resolution (the exact class hono-preact breaks on), and dev/prod CSS parity by inlining all styles in dev. Two corrections from adversarial verification temper that credit: SK's font Link headers are opt-in only (the default preload filter is js/css, `respond.js:50`), so end-to-end font preloading requires explicit user action in both frameworks, and SK's uncapped Link header (no budget, set unconditionally when non-empty) is a real exposure ours budgets at 12KB against Cloudflare's ~16KB cap. Structurally SK gets per-route CSS free from per-node code splitting (`find_deps` harvests `chunk.css` off the manifest), which is precisely why hono-preact's post-hoc monolith splitter needed, and got, proportionately heavier guarding; SK's `find_deps` itself has no direct unit test and its emission half is embedded in the ~630-line `render_response` function. React Router unit-tests the adjacent route-chunking half well (1,635-line test) but its manifest asset derivation lives untested inside the 3,892-LOC plugin, it emits no Link header or Early Hints (head emission is user-rendered `<Links/>`/`<Scripts/>`), and its runtime links layer carries `any`-typed predicates and descriptor casts; in its favor, RR does mitigate dev-mode CSS FOUC via dev-only critical-CSS inlining (`unstable_getCriticalCss`, `plugin.ts:823`), so two of the three peers address the dev CSS gap hono-preact defers to #258. SolidStart's asset path is entry-only head assets, module-level cross-request caches, `as any` throughout, and no tests.

## Cross-cutting themes

**Recurring strengths that hold up under verification.** All seven subsystems verified ours-better overall at v0.10.1 (June: six of six, with realtime peerless). Four patterns recur:

1. **The backlog gets executed, and verification confirms it.** This is the strongest meta-signal in the rerun. The June report's recommendations were not roadmap decoration: the P0-class socket resolver is required and construction-validated on all three handlers with executed fail-closed tests; the pump got its direct harness; the reader-construction duplication became five per-mode factories sharing one `settle()`; the sockets god-file was split exactly as prescribed with the six bangs narrowed away; the missing type-level suites (compose-server-chain, stream-registry) exist and run clean; the realtime cross-runtime traps became compile errors; loader validation-issue parity closed end to end with e2e assertions on both loader failure paths. Of the June recommendation list, two sub-items were not executed: the `server-entry.ts` diagnostics extraction (which regressed instead) and recommendation 7's deny-path payload typing (the accessor centralized the casts, but neither a `TError` generic nor a user-guard-forcing typed accessor exists, per the data ledger).
2. **One engine, many drivers, with contracts pinned in multiple representations.** The June "single engine" discipline held and deepened: one join (`joinRoutePath`) mirrored by the type-level `Here` and shared with the server boot guard via `subtreePatternOf`, so the #272 bug class is closed by construction; the streaming wire dual-locked between producer and consumer modules plus a type-level pin; the room protocol once behind `RoomTransport`; one chain engine, one composition home, one construction guard; one AST parser and one export allowlist in the build layer; one schema-coercion module consumed by all three loader execution paths.
3. **Fail-closed security posture, layered and adversarially tested.** Boot-time route-binding guards (exact, subtree, registry), byPattern-only guard resolution with the URL-fuzzy mechanism deleted, construction asserts on all three handlers, `composeServerChainOrFailClosed` single-sourcing the route-bound invariant, prod error masking defaulting closed at five layers on both streaming wires, fail-closed unstubbable dynamic imports at build, and the realtime `x-hp-*` strip. Each layer carries attacker-model tests (attacker-chosen POST URL, registry-fallback rescue attempt, guard-exactly-once-on-deny, header spoof/smuggle), not happy paths. The one confirmed hole in this story is the serverRoute socket/room binder (below).
4. **Test posture: fast adversarial unit-first over the riskiest code, plus type-level suites the peers mostly lack, plus escalating integration tiers.** Every "ran green" claim in this report was executed during the audit (routing 123 tests/352ms, nav/transition 85/527ms, middleware 56 across the eight core suites/399ms, SSR pump/shell/runner suites, realtime 78/321ms, build 112/381ms plus createServerEntry 8/316ms, assets 135 across 12 files). Predicate-over-cast discipline with sanctioned-boundary comments recurs everywhere; the typed-routes engine and the middleware engine remain literally cast-free.

**Recurring weaknesses that survive at v0.10.1.**

1. **Hotspots regrow after extraction.** The pattern across four subsystems: the prescribed extraction lands, then the next feature wave accretes inside the same closure or file. `define-routes.tsx` 624 to 710 LOC across four PRs after the fold; `useAction`'s mutate callback back up to ~300 of 696 lines (June: 267 of 522) because the #221 schema gate and onMutate plumbing landed inside it; `use-loader-runner.tsx` 344 to 521 LOC absorbing the ADT rewrite; `server-entry.ts` 349 to 486 LOC gaining a fourth concern; `renderPage` re-accreted a ~50-line inline asset-hint policy block (`render.tsx:194-246`); `page-middleware-host.tsx` 318 to 367. None is a god-file by peer standards (the peers' comparable cores are 3,451 to 7,722 LOC), but the growth vector is consistent and unmanaged.
2. **Parity-by-comment instead of shared code.** Loader/action invocation assembly is triplicated across the RPC handler, the SSR runner, and the server caller, aligned by "cannot diverge" comments; the two HTTP handlers carry twin `EMPTY_PAGE_USE`/TimeoutError/500-shaping blocks; realtime duplicates its registry folds and frame decode; the vite package byte-copies iso path helpers and twin-declares the manifest types. Every instance is individually small; collectively it is the codebase's largest un-mitigated drift surface, because none has a compile-time tie.
3. **New API arms shipping ahead of their runtime semantics or their documentation.** `serverRoute(r).socket`/`.room` silently discard the route argument (`server-route.ts:201-202`) on the auth-relevant path, with no boot error or dev warning, while the binder's docstrings promise route binding; the loader RPC deny envelope drops the typed `DenyCode` the action envelope carries (`outcome-translation.ts:64-74`); deny remains second-class on the imperative surface (`ActionResult` deny `data?: unknown`, plain-Error failure arm on `mutate`); liveStream's contextual-typing wart is documented only in a type-test comment; Vite `base` breaks route CSS with no guard; the dev route-CSS FOUC divergence is silent.
4. **Guarded/unguarded asymmetries between siblings.** `eventStream`'s unbounded queue beside the hooks' capped, warning queue; the never-truncated font Link part beside the budgeted closure; the silent module-global `fallbackStore` degrade when `node:async_hooks` is absent (`cache.ts:139,151`) beside the loud failure taxonomy everywhere else; `sse.ts`'s `step.value !== undefined` presence check beside the ADT that banned exactly that heuristic.
5. **Small recurring cast-doctrine lapses.** Individually trivial, but the same shape in five subsystems: `route-active.ts:52` and `define-routes.tsx:22` (uncommented boundaries), `loaders-handler.ts:313` (`routeId!` where the sibling handler solved it cast-free), `loader-runner-server.ts:89-103` (the defineProperties double cast), `streaming-ssr.ts:21,39` (untyped ALS reads), `realtime-do.ts:303,327,356` (negative-space-as-room), `guard-strip.ts:65`/`server-loader-validation.ts:38` (redundant post-narrow), `preload-manifest.ts:224`.
6. **Render-phase side effects in `use-loader-runner.tsx`** (subscribe, abort, cache writes during render behind ref bookkeeping, :301-489): correct under the documented SSR single-render invariant and now directly harnessed, but the framework's most Preact-semantics-coupled code and the first casualty of any future concurrent-rendering change.

## Recommendations

Priority-ordered, highest-impact first. Every item is verified still-open against the gap ledgers at HEAD 60df3c21; nothing already shipped is recommended.

1. **Close the `serverRoute(r).socket`/`.room` binder hole (P0-class contract/auth).** Two of the binder's four arms silently discard the route argument (`server-route.ts:201-202`), so a `serverRoute('/admin').socket(...)` in a src/server registry module resolves to `SOCKETS_RPC_PATH` and runs with no page-tier gates, and `route-binding-guard`'s CONTAINERS covers only serverLoaders/serverActions, so neither boot error nor dev warning fires. Either make the arms actually bind (stamp a `__routeId`, extend the boot guard's containers and the socket path resolver to honor it) or remove/throw on the arms so the API cannot promise gates it does not run. This is the one remaining fail-closed-discipline hole in an otherwise defense-in-depth story.
2. **Validate `use` entries at construction.** `partitionUse` buckets anything whose `__kind !== 'middleware'` into observers with no runtime check (`use-partitioner.ts:18-21`), so a malformed entry read structurally off a user module becomes a silently-dropped middleware, the exact failure class everything else fails closed on. Add an isMiddleware/isObserver validation (throw on neither) at `composeServerChain`. June gap, still open.
3. **Deny-path parity and typing.** Carry `code` in the loader RPC deny envelope (`outcome-translation.ts:64-74`) as the action envelope already does; surface deny on `mutate`'s imperative result (a typed failure arm) so callers stop needing the `useActionResult` pairing to switch on `DenyCode`; and thread a type (or a guarded accessor forcing a user narrow) through `ActionResult`'s deny `data?: unknown` (`use-action-result.ts:20`), the one June data-layer gap still only partially closed.
4. **Guard Vite `base` at config time.** Same silent-broken-page class the `cssCodeSplit` guard already covers (`client-entry.ts:118-124`); today a configured base 404s render-blocking route CSS with no diagnostic (`route-preload.ts:359-366`). A hard error is one afternoon; real base support can follow.
5. **Single-source the comment-synced cross-package contracts.** (a) Move the preload artifact types (`RoutePreloadMap`, `PreloadArtifact`/`PreloadManifest`) into the shared iso contract module both packages already import constants from; (b) execute TODO(#249): extract the byte-copied path helpers into a preact-free iso leaf, or at minimum add a parity test; (c) replicate the iso door's exact-set export test on the server and internal-cloudflare doors so a door rename fails the fast unit tier, not just gated workerd runs.
6. **Cap or dev-warn the unguarded queues and degrades.** `eventStream`'s unbounded FIFO (`event-stream.ts:47`) should mirror the ws-lifecycle 128-cap-plus-warn; the `createCache` ALS-absent fallback to a module-global store (`cache.ts:139,151`) deserves a one-time boot warning; consider a warn when the never-truncated font Link part alone exceeds the header budget.
7. **Fill the named unit-test gaps in otherwise well-covered code.** `expandGlobFs` (the only fs-touching hint-discovery path, zero direct tests), `resolveRoomKey`'s `:x?`/`:x*`/`:x+` branch, `deriveLayoutLocation` (and document or fix its hardcoded `'rest'`/`'0'` param drop at `define-routes.tsx:530`, which would silently swallow a user param literally named `:rest`), and `server-loader-validation`'s missing `export class` collection.
8. **Decompose the regrown hotspots, reusing the codebase's own extract-and-share pattern.** In risk order: extract `renderPage`'s inline asset-hint policy block (`render.tsx:194-246`) into a module beside `preload-modules.ts`; move `server-entry.ts`'s two AST diagnostics beside the parser (the one June recommendation never executed) and re-home the probe watcher; extract the #221 schema gate and the outcome-recording protocol out of `useAction`'s mutate callback; and consider one shared loader/action invocation engine to replace the three parity-by-comment assembly sites.
9. **Cast-hygiene sweep.** One small PR: `routeId!` (`loaders-handler.ts:313`, copy the actions handler's cast-free derivation), the `ServerLoaderCtx` defineProperties double cast, typed ALS store accessors in `streaming-ssr.ts`, an `isRoomConnAttachment` predicate for `realtime-do.ts`'s negative space, the two redundant post-narrow casts in vite, `(e as Error)` in `preload-manifest.ts`, and sanctioned-boundary comments on `route-active.ts:52` / `define-routes.tsx:22`.
10. **Polish (low-risk, one batch).** Delete the dead `preResolved` parameter (`sockets-handler.ts:54-56`); dedupe `isAsyncGenerator` and re-home `maskStreamError`; key-sort `useRoom`'s key serialization; parameterize the three remaining stub-template factories; retire `SseGeneratorOptions` and the unused `_c: Context` params; document the liveStream contextual-typing workaround in the user-facing docstrings; add the one-time dev notice for the route-CSS FOUC divergence (#258 interim); per-value string validation in `validateLocation`'s pathParams (matching `resolveRoomKey`).

## Methodology & caveats

**Pipeline.** This rerun used the same multi-agent profile, judge, verify, synthesize pipeline per subsystem as the June report, now over seven subsystems (asset delivery is net-new). A profiling pass read hono-preact source at HEAD 60df3c21, mapped every 2026-06-24 gap onto the renamed/split current files (sockets-handler split, the new create-server-entry/route-binding-guard/loader-state/route-change/route-preload/css-auto-split modules), and measured rather than estimated (LOC, cast inventories, mutable-binding counts, test executions). A judging pass assigned per-lens verdicts against the strongest peer with paired ours/theirs citations. An independent verification pass then re-read every load-bearing citation in both codebases, re-ran the referenced test suites, re-measured every quoted peer line count, and classified each claim supported / imprecise / unsupported. Where the verifier's adjusted verdict differs from the raw judge verdict, this report follows the verified verdict; unsupported claims were dropped and imprecise ones corrected in place (examples that survive in corrected form: RR's integration directory holds 87 test files among 114 .ts files, not 114 tests; hono-preact's composition root assembles 11 plugins, not 12; `createServerEntry` has 8 direct tests, not 10; the realtime test-LOC accounting was rebuilt from scratch). One scope note on the verification tier: verifiers recorded lens-level verdict adjustments only where they changed a cell (middleware complexity/tests, assets API design); for the other five subsystems the verifier endorsed the judge's lens cells wholesale (verdict holds) after re-deriving the material claims beneath them, so those cells carry claim-level rather than cell-by-cell re-verification.

**Grass-is-greener audit.** The verification tier explicitly hunted for pro-hono-preact claims that would not survive reading peer source, and found two this run. First, the judge's claim that React Router "has no type-level tests" was false: RR ships roughly 50 inline `Expect<Equal<>>` compile-time assertions, including 21 pinning the single-fetch data channel's serialization typing; the narrower true statement (RR does not type-pin its streaming wire internals and keeps `data: any`/expando Promises where hono-preact pins the wire union in dedicated suites) narrows the SSR type-safety edge without flipping it. Second, crediting SvelteKit with default font Link-header preloads overstated it: SK's default preload filter is js/css-only, so font preloading is opt-in there, and both frameworks require explicit user action for fonts. Both corrections are folded into the sections above. No pro-hono-preact verdict collapsed on reading competitor source.

**Caveats.**

- *Maturity and scope normalization.* React Router, SvelteKit, and SolidStart are far more mature and carry features hono-preact lacks (RSC, single-fetch, fetchers/revalidation, SPA mode, a large adapter ecosystem). Each lens judged only the comparable concern in each subsystem; hono-preact was not credited for features the peers lack, peers were not penalized for complexity that serves features they ship, and where a peer avoids an error class by construction (RR/SK colocate loaders with routes, so route misbinding cannot exist there), hono-preact's guard was credited as parity restoration for its composition model, not as a peer gap. hono-preact's small size is also an advantage in keeping seams clean; the verdicts describe current code quality, not trajectory.
- *SvelteKit is JS + JSDoc, not TS.* Its runtime type safety was judged qualitatively (JSDoc discipline, cast-justification comments, its genuinely strong generated types and type-level test files), never by a raw cast count, and it was never penalized merely for being JS.
- *Competitors were freshly re-cloned at current main for this rerun*, unlike the June rerun which reused the original checkouts: react-router 8.2.0 @ ac5d9d5 (bumped from the 8.0.1 the prior reports profiled), @sveltejs/kit 2.69.2 @ 5c38e51 (from 2.67.0), and @solidjs/start 2.0.0-alpha.3 @ ec1b82b (same alpha line, newer commit). Every peer number cited here was re-measured at these commits rather than reused (e.g. RR `router.ts` is 7,722 LOC and `plugin.ts` 3,892 at 8.2.0; SK `client.js` 3,451 and `vite/index.js` 1,718 at 2.69.2). The version bumps did not fix any weakness this report relies on, and where a peer improved (SvelteKit's remote-functions layer, including `query.live` and its 88-negative type suite), the improvement is credited in the relevant section.
- *Realtime's no-peer caveat is updated, not retired.* React Router and SolidStart still ship no first-party realtime at these versions (grep-verified). SvelteKit's experimental, default-off `query.live` is a partial comparator on the one-way SSE streaming slice only; that slice was compared like-for-like, while the duplex sockets / rooms / presence / pub-sub majority of the subsystem remains judged intrinsically against the framework's own bar, with the explicit understanding that the absence of a peer is not itself a quality result.
- *Known under-examined slices.* Three comparable concerns were noticed but not judged this run and are left to a future rerun: navigation prefetch (hono-preact's opt-in Speculation Rules emitter, `speculation-rules.ts`, vs SvelteKit's `data-sveltekit-preload-*` vs React Router's `<PrefetchPageLinks>`); the server-side caller slice (`ctx.call`/`createCaller` vs invoking SvelteKit remote functions server-side, including how each treats page-tier guards on internal calls, which matters because this report flags `ctx.call`'s page-tier bypass as a hono-preact gap without checking the peer analog); and the streaming `createCaller` path, which was confirmed present but produced no findings either way.
- *Everything executable was executed.* Test-suite results quoted in this report (green runs, counts, durations) come from runs performed during this audit on 2026-07-11, not from CI history or PR descriptions; gap closures were accepted only on positive evidence in current source or executed tests.
