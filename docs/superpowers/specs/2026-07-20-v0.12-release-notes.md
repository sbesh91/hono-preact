# hono-preact v0.12.0

A **correctness, security, and packaging** cut. No new public API to learn: this
release closes two P1 holes (route-scoped action authorization and Cloudflare
WebSocket upgrades), fixes progressive-enhancement forms and several SSR and
component defects, and cleans up the published artifact and the scaffold pins.
It ships in lockstep with `create-hono-preact`; `hono-preact-ui` did not change
and stays `0.3.0` (a framework-only cut).

## Versions

| Package | From | To | Why |

## Behavior changes to review before upgrading

No public API was removed or renamed, but two behaviors changed. Neither should
affect a correctly-built app; both close a gap where the previous behavior was
wrong.

### Route-scoped actions are now authorized against their route (#288)

Server actions bound to a route via `serverRoute(r).action(...)` now run their
route's page-tier guards, because the action **guard** context
(`ServerActionCtx`) finally carries a route-authoritative `location`. Before
this, a route's middleware could not gate its actions: the guard ctx had no
`location`, so a guard keyed on route params or path could not tell which route
the action belonged to, and an action POST that did not match the route was
allowed through.

- The `location` is derived server-side from the POST URL matched against the
  action's own `routeId`, not from anything the client sends, so it cannot be
  spoofed.
- An action POST whose URL does not resolve to the action's route is now
  **denied** with a 403 before the guard chain runs. An in-route POST that a
  route guard rejects returns that guard's deny status. Either way, if your app
  was (intentionally or not) invoking a route-bound action from a URL outside its
  route, that call no longer runs. This is the fix, not a regression: such a call
  was previously bypassing the route's guards.
- Bare `defineAction(...)` (not route-bound) is unaffected; it has no route
  scope to authorize.

### An SSR loader `deny()` now renders the route's `errorFallback` (#287)

When a page loader throws `deny(status, message)` during SSR, the response used
to short-circuit to a bare text body at the deny status (no document shell, no
styles). A deep link or crawler saw unbranded text.

Now, if the denying loader declares a local `errorFallback`, the framework
renders that fallback into a **full document at the deny status**, matching what
client-side navigation already showed, and bakes a hydration marker so the
client adopts the error UI without a wasted refetch. A loader that denies with
**no** local `errorFallback`, a middleware-scope `deny()`, and a loader
`redirect()` are unchanged (bare text at the status, and a real 302
respectively). Deny-supplied response headers now ride the document on both the
buffered and streaming response paths.

## New capability

### `upgradeWebSocket` works on the Cloudflare adapter (#291)

`upgradeWebSocket` was Node-adapter-only; on the Cloudflare adapter it threw at
request time. The Cloudflare adapter now has a framework-owned upgrader built on
`WebSocketPair`, at parity with the Node path: `onOpen` fires and is guarded,
the `Upgrade` header check is case-insensitive, and `ws.url` is the request URL.
The upgrade seam is independent of the realtime connector, so a raw
`upgradeWebSocket` handler works without pulling in a Durable Object.

## Fixes

### Forms and SSR

- **Progressive-enhancement forms actually work (#292).** An SSR-rendered
  `<Form>` was emitting **empty** `__module` / `__action` hidden inputs, so a
  no-JS or pre-hydration native POST reached the server as action `''` and got
  "Action '' not found". The Vite plugin now threads the module/action keys into
  `serverActions` (mirroring the existing `serverLoaders` path), so the SSR
  action ref carries its keys and `<Form>` renders populated inputs. No-JS and
  pre-hydration submits now hit the right handler.

### Components

- **`NavLink` `aria-current` is suppressable (#295).** `NavLink` computed
  `aria-current` such that a caller could not turn it off (passing
  `aria-current={undefined}` on an active link still emitted `page`). It now uses
  a presence check: an explicit `aria-current={undefined}` suppresses the
  attribute, while an omitted prop still computes `page` when active. (Active
  detection remains path-only.)
- **Dev warning for an inert `view-transition-class` (#294).** A
  `view-transition-class` with no accompanying `view-transition-name` is silently
  inert; the framework now warns in development so the mistake is visible. The
  warning is dev-only and does not ship in production builds.

### Packaging and scaffold (from the v0.11 tarball review)

- **Published tarball no longer carries dangling `.d.ts.map` (#279).** The
  `hono-preact` `files` list now excludes `dist/**/*.map`; those 10 declaration
  maps pointed at `src/` that the tarball never shipped, so they were dead
  references. Workspace-local go-to-definition is unaffected.
- **Scaffolded apps pin `preact-iso` immutably (#280).** Scaffolds pinned
  `preact-iso` to a mutable git branch (`github:preactjs/preact-iso#v3`); new
  apps now get an immutable pin so a scaffold is reproducible.
- **Published manifest no longer lists unpublished workspace devDeps (#281).**
  The published `hono-preact` manifest carried devDependencies on the private,
  unpublished `@hono-preact/*` packages; those are trimmed.

### Dev tooling and docs

- **Docs-site Vite aliases are generated from package exports (#290).** The
  hand-maintained alias list in `apps/site/vite.config.ts` missed subpaths (a
  missing subpath resolved to `dist/`, which workerd refuses, killing dev). The
  list is now derived from each package's `exports` map and is a strict superset
  of the old one. This is a docs-site dev fix; it does not affect consumer apps.
- **`.server` export whitelist is now guarded by a fast unit test (#289).** The
  build-time whitelist enforcement already existed (`serverLoaderValidationPlugin`);
  a violation was only caught late, during the CI site build. A new unit test runs
  the same check over every `.server.*` file so a violation now fails `pnpm test`.
- **Loader `timeoutMs` docs clarified as RPC-only (#286).** `timeoutMs` is
  enforced on the loader RPC path (client navigation) only; it is **not** applied
  during the initial SSR render. The loader docs and `defineLoader` JSDoc now say
  so. Behavior is unchanged; the docs previously left it ambiguous.
- **Internal batch cleanups (#304).** Build-tooling and test-guard corrections
  (the client-size probe now measures true consumer-facing size by defining
  `import.meta.env.DEV = false`, plus loader-parser and vitest-config tidy-ups).
  No consumer-facing runtime change.

## Upgrade

Bump the dependency and reinstall:

```jsonc
// package.json
"hono-preact": "^0.12.0"
```

There is nothing to migrate. Review the two behavior changes above: if an app
relied on invoking a route-bound action from outside its route (it should not
have), that call is now denied; and an SSR loader `deny()` now renders the
route's `errorFallback` when one is declared.

`create-hono-preact@0.12.0` scaffolds against `hono-preact@^0.12.0` and pins
`preact-iso` immutably. `create-hono-preact` requires Node `^22.18.0 || >=24.11.0`.
