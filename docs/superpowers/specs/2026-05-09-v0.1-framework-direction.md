# v0.1 Framework Direction

**Date:** 2026-05-09
**Status:** Draft
**Lens:** Zero users. Pre-announcement. No npm version yet. Design for the day someone first opens the README, not for hypothetical migration.

## TL;DR

`hono-preact` ships at v0.1 as a small, opinionated full-stack framework. Hono on the edge, Preact in the browser, **code-defined routes**, typed RPC, streaming everywhere. No filesystem-routing magic, no config file the user has to read.

The user authors six things, total:

1. `src/Layout.tsx` — HTML shell.
2. `src/routes.ts` — the route table, as a data structure.
3. `src/views/**` — view components, organized by domain (the user's call).
4. `src/views/*.server.ts` — paired server bundle (optional, declared explicitly in `routes.ts`).
5. `src/api.ts` — optional, only if there are non-framework Hono routes.
6. `vite.config.ts` — one line: `defineApp(...)`.

Everything else (`client.tsx`, `iso.tsx`, the `process` shim, the `import.meta.glob` calls in `server.tsx`, the manual `<script>` branching, the alias block) is owned by the framework and disappears from user code.

The pitch sentence:

> `hono-preact` — a small full-stack framework. Hono on the edge, Preact in the browser, code-defined routes, typed RPC, streaming everywhere. No config, no filename magic.

"No filename magic" is a deliberate position against filesystem-routed frameworks. Some users will pick this *because* of it.

## 1. Code-defined routes (`defineRoutes`)

**Why not filesystem routing.** Filesystem routing optimizes for the framework author at the cost of the user. URL design becomes filename design, refactors churn git history, every route gets a folder whether it earned one, layouts and groups become magic prefixes. With code-defined routes the user organizes files by domain, refactors URL paths in one edit, and reads the entire route table at a glance.

**Shape.** The route table is a data structure, not JSX:

```ts
// src/routes.ts
import { defineRoutes } from 'hono-preact';

export default defineRoutes([
  { path: '/',           view: () => import('./views/home') },
  { path: '/movies',     view: () => import('./views/movies'),  server: () => import('./views/movies.server') },
  { path: '/movies/:id', view: () => import('./views/movie'),   server: () => import('./views/movie.server') },
  { path: '/watched',    view: () => import('./views/watched'), server: () => import('./views/watched.server') },
]);
```

`view` and `server` are independently lazy (separate code-split chunks). The framework owns `__loaders` and `__actions` dispatch from this table. No glob, no `import.meta.glob` in user code, no implicit pairing by sibling filename.

The route table *is* the manifest. Devtools, agents, and the framework itself can read it. "What handles `/movies/:id`?" is one open and one ctrl-F.

**Replaces:**
- `iso.tsx` (the `<Router>` + `<Route>` JSX table).
- `actionsHandler(import.meta.glob('./pages/*.server.ts'))` and `loadersHandler(...)` in `server.tsx`.
- The implicit "`.tsx` and `.server.ts` must share a basename" convention.

### Nesting

Nesting is for **layout composition**, not for parallel loaders. We deliberately do not build a parallel data graph (out of scope at v0.1). That makes nesting much simpler than Remix's: only one loader runs per request, layouts compose via children.

#### Shape

A route may declare `children`. Each child's `path` is relative to its parent's `path`. The parent's `view` becomes a wrapper around whichever child matches.

```ts
defineRoutes([
  { path: '/', view: () => import('./views/home') },
  {
    path: '/movies',
    view: () => import('./views/movies-layout'),
    children: [
      { path: '',     view: () => import('./views/movies-list'), server: () => import('./views/movies-list.server') },
      { path: ':id',  view: () => import('./views/movie'),       server: () => import('./views/movie.server') },
    ],
  },
  { path: '*', view: () => import('./views/not-found') },
]);
```

URLs:
- `/movies` matches the parent + the empty-path child → renders `<MoviesLayout><MoviesList /></MoviesLayout>`.
- `/movies/123` matches the parent + the `:id` child → renders `<MoviesLayout><Movie /></MoviesLayout>` with `pathParams.id === '123'`.

#### URL composition rules

| Rule | Reason |
|---|---|
| Child path appends to parent path with a `/` | Standard, predictable. |
| Empty child path (`''`) is the "index" — matches parent's path exactly | Lets you render a default child without an extra URL segment. |
| Child paths must not start with `/` | Avoids ambiguity about whether they're absolute or relative. Plugin error at build time if violated. |
| `*` as a path matches anything not matched by siblings | Works at any nesting level. |
| Routes match in source order | First match wins, including catchalls. |
| Trailing slashes normalised | `/movies/` and `/movies` are the same route. |

#### Layout pattern: children prop

A parent view receives the matched child tree as a prop. No `<Outlet />` context lookup, no magic — just children.

```tsx
// src/views/movies-layout.tsx
import type { ComponentChildren } from 'preact';

export default function MoviesLayout({ children }: { children: ComponentChildren }) {
  return (
    <section class="p-1">
      <header><a href="/">home</a></header>
      <main>{children}</main>
    </section>
  );
}
```

This matches the existing `definePage` Wrapper ergonomics. No new mental model.

#### Data model: one loader runs

Each route may have a `server` import. **The matched leaf's loader runs. Parent loaders do not.** This keeps the v0.1 promise of no parallel data graph.

- A route with `children` should not declare `server`. The plugin warns at build time if it does.
- Children that don't need data simply omit `server`.
- Cross-route invalidation still works via loader refs:
  ```tsx
  const toggle = useAction(actions.toggleWatched, {
    invalidate: [moviesListLoader, watchedLoader],
  });
  ```

If you want a parent to have data, restructure: hoist the parent into a sibling layout pattern, or pull the data-needing parent into the child. v0.1 deliberately doesn't try to coordinate ancestor + descendant loaders — that's where most nested-routing complexity comes from in other frameworks.

#### Errors and fallbacks

Each route can declare `fallback` (Suspense) and `errorFallback` (error boundary), same as today's `definePage`.

```ts
{
  path: '/movies',
  view: () => import('./views/movies-layout'),
  errorFallback: <p>Movies are down.</p>,
  children: [
    { path: '',    view: () => import('./views/movies-list'), fallback: <p>Loading list…</p> },
    { path: ':id', view: () => import('./views/movie'),       fallback: <p>Loading…</p> },
  ],
}
```

Errors bubble to the nearest ancestor with an `errorFallback`. The leaf's `fallback` handles its own Suspense; the parent layout chrome stays mounted.

#### Lowering to `preact-iso`

`preact-iso` has a flat route table. The framework flattens at module-eval time:

1. Walk the nested tree, producing a list of leaves with their full path and ancestor chain.
2. For each leaf, generate `(props) => <Ancestor1><Ancestor2><Leaf {...props} /></Ancestor2></Ancestor1>`.
3. Register each leaf as a flat `<Route path component>` with `preact-iso`.

Users never see the flat form. The lowering is internal. Devtools and the route manifest show the nested table.

#### Out of scope at v0.1

| Excluded | Why | Workaround |
|---|---|---|
| Pathless routes (layout grouping with no URL segment) | Adds complexity, real users haven't asked yet | Repeat the layout component as a regular import in each child view. |
| Parallel parent + child loaders | Explicitly out of scope for v0.1 | Hoist the loader to whichever route actually owns the data. |
| Loader composition / `useParentLoaderData` | Same | Same. |
| Per-segment server-only middleware beyond guards | Out of scope | Use Hono `.use()` in `api.ts` for path-prefix middleware. |

Reasonable post-v0.1 additions once real users tell us what shape they need.

#### Migrating the demo's `/movies` routes

Today (`iso.tsx` plus a single `movies.tsx` doing double duty):

```tsx
<Route path="/movies"   component={Movies} />
<Route path="/movies/*" component={Movies} />
// Movies renders header + list + an inner <Router> for /:id → Movie
```

After:

```ts
// routes.ts
{
  path: '/movies',
  view: () => import('./views/movies-layout'),
  children: [
    { path: '',    view: () => import('./views/movies-list'), server: () => import('./views/movies-list.server') },
    { path: ':id', view: () => import('./views/movie'),       server: () => import('./views/movie.server') },
  ],
}
```

`movies-layout.tsx` keeps the chrome (header, "home" link, watched-count badge if it should always show). `movies-list.tsx` is the previous `Movies` body without the header and without the inner `<Router>`. `movie.tsx` is unchanged. The single double-purposed `Movies` component splits into a layout and a list — a clean responsibility split.

The two `<Route>` entries become one nested entry. The inner `<Router>` in user code disappears. The route table reads top-down.

## 2. Typed bindings as methods on the ref

Without filesystem routing or virtual modules, the typed-bindings story rests on the loader/action refs themselves carrying their types through and exposing methods.

**Loader data:**

```tsx
// src/views/movies.tsx
import { loader } from './movies.server.js';

function Movies() {
  const { movies } = loader.useData();   // typed end-to-end
  return <ul>{movies.map(m => <li>{m.title}</li>)}</ul>;
}
```

`loader.useData()` replaces `useLoaderData<typeof loader>()`. The hook is a method on the ref; its return type flows naturally. No `typeof` gymnastics, no virtual module, no codegen.

**Cache invalidation:**

```tsx
import { loader as moviesLoader } from './movies.server.js';
import { loader as watchedLoader } from './watched.server.js';

const { mutate } = useAction(serverActions.toggleWatched, {
  invalidate: [moviesLoader, watchedLoader],
});
```

`cacheRegistry` and string-keyed names disappear. Cross-route invalidation works because the loader ref is a value you import. Type-checked, rename-safe, no global registry.

**Actions** keep working as today (`useAction(stub, opts)`) and additionally expose a method form:

```tsx
const toggle = serverActions.toggleWatched.useAction({ invalidate: [moviesLoader] });
await toggle.mutate({ movieId: 1, watched: true });
```

Both forms ship; pick whichever reads better in context.

**Replaces:**
- `useLoaderData<typeof loader>()` pattern.
- `cacheRegistry` (deleted, including `.register`/`.invalidate`/`.acquire`).
- The `'movies-list'`-style string keys passed to `createCache`.

## 3. `defineApp()` for `vite.config.ts`

`vite.config.ts` collapses to:

```ts
// vite.config.ts
import { defineApp } from 'hono-preact/vite';

export default defineApp({
  layout: 'src/Layout.tsx',
  routes: 'src/routes.ts',
  plugins: [/* user vite plugins, e.g. @mdx-js/rollup */],
});
```

Internally `defineApp` composes everything `honoPreact()` does today plus preact, the dev-mode resolve aliases (currently hand-authored), and the wrangler glue. **MDX (and any other content tooling) stays in userspace** — the user adds `@mdx-js/rollup` to the `plugins` list themselves, exactly as `apps/app` does today. The framework does not own a content primitive at v0.1.

Advanced users can still drop into a normal Vite config; `defineApp` is the default path.

**Replaces:**
- The 60-line `vite.config.ts` in `apps/app`, including the alias block and the visualizer toggle.

## 4. Framework-provided client entry

`client.tsx` is boilerplate. The framework ships its own and references it via a virtual module:

```ts
// virtual:hono-preact/client (owned by @hono-preact/vite)
import { hydrate } from 'preact';
import { LocationProvider } from 'preact-iso';
import { Router } from '<framework-internal>';
import routes from '/src/routes';

const App = () => (
  <LocationProvider>
    <Router routes={routes} />
  </LocationProvider>
);

hydrate(<App />, document.getElementById('app')!);
```

The `process` shim is **not** an explicit import in this source. It is injected by the existing `clientShimPlugin` transform, which prepends `import 'virtual:hono-preact/client-shim';` onto whatever the configured client entry is. When the framework provides its own entry, that virtual module simply *becomes* the configured entry and gets the same transform pass. The shim machinery never appears in any source the user reads or the framework author maintains; it lives entirely in the plugin.

`Layout.tsx` references it through a framework component:

```tsx
import { ClientScript, Head } from 'hono-preact';

export default function Layout({ children }) {
  return (
    <html>
      <Head defaultTitle="hono-preact" />
      <body>
        <main id="app">{children}</main>
        <ClientScript />
      </body>
    </html>
  );
}
```

`<ClientScript />` resolves to `/static/client.js` in prod and the dev virtual module in dev. The user never writes the branching.

**Replaces:**
- `apps/app/src/client.tsx`.
- The `import.meta.env.PROD ? <script src="/static/client.js"> : <script src="/src/client.tsx">` ternary in `Layout.tsx`.
- Hand-managed `hoofd/preact` wiring in `Layout.tsx` (now wrapped by `<Head>`).

## 5. Streaming loaders + `<Form>` parity

Today: actions can stream (`bulkImportWatched` returns a `ReadableStream`); loaders cannot. `<Form>` calls `response.text()` then `JSON.parse` so it dies on streaming actions.

Both are fixed:

```ts
// loaders can stream too
export const loader = defineLoader<ReadableStream<{ progress: number }>>(async () => { ... });
```

`<Form>` and `useAction` share dispatch internally. Both handle streaming actions identically. Existing punch list (`docs/design-concerns-2026-04-25.md` items 1, 2, 5, 7) gets cleared in the process.

This is the differentiator. Most frameworks don't do streaming actions cleanly, never mind streaming loaders. It's the reason for someone to pick this framework over Remix or TanStack Start.

## 6. MDX and content collections stay in userspace

**Out of scope at v0.1.** The framework does not ship a content-collection primitive (`defineCollection`, frontmatter typing, automatic nav). Users wire MDX exactly as `apps/app` does today: add `@mdx-js/rollup` to their `defineApp({ plugins })`, mount a single view at `/docs/*` in `routes.ts`, and use `import.meta.glob` inside that view to enumerate content.

```ts
// src/routes.ts
defineRoutes([
  // ...
  { path: '/docs/*', view: () => import('./views/DocsRoute') },
]);
```

`DocsRoute.tsx` continues to own its inner `<Router>`, the MDX glob, and its `<article>` wrapper. The framework's job is to make sure `defineRoutes` and the rest of the primitives compose cleanly with any pattern a user invents on top, not to ship a particular pattern.

This keeps the v0.1 surface focused on routing, data, and actions. A first-class collections primitive is a fine post-v0.1 addition once we have real users telling us what shape it should take.

## 7. Single guards list + ActionGuardError status type fix

`definePage` collapses `serverGuards` and `clientGuards` into one ordered `guards` list. Each guard is built by an env-specific factory:

```ts
import {
  defineServerGuard,
  defineClientGuard,
  definePage,
} from 'hono-preact';

const adminGuard = defineServerGuard(async (ctx, next) => {
  if (!await isAdmin(ctx)) return { redirect: '/forbidden' };
  return next();
});

const scrollRestore = defineClientGuard(async (_, next) => next());

definePage(Admin, { guards: [adminGuard, scrollRestore] });
```

There is no unified `defineGuard` and no `runs: 'both'`. A user who needs the same body on both sides writes `[defineServerGuard(fn), defineClientGuard(fn)]`.

The Vite plugin (`guardStripPlugin`) rewrites `defineServerGuard(...)` to a passthrough stub in the client bundle, and `defineClientGuard(...)` to a passthrough stub in the server bundle. Server-only helpers referenced only inside a stripped body tree-shake out. The runtime ALSO filters opposite-env guards out of the execution chain, so the stub never runs anyway. Double safety: bundle stripping at build time, chain filtering at runtime.

`ActionGuardError`'s constructor status type tightens from `number` to Hono's `ContentfulStatusCode`. The narrow cast at the response boundary (`as 400 | 401 | 403 | 404 | 429 | 500`) goes away. `ContentfulStatusCode` is re-exported from `hono-preact` for ergonomics.

The full design lives at `docs/superpowers/specs/2026-05-13-single-guards-list-design.md`.

## 8. One published package, three subpaths

Today: `@hono-preact/iso`, `@hono-preact/server`, `@hono-preact/vite` are workspace packages and the umbrella `hono-preact` package re-exports them but is unused.

Future:

```ts
import { definePage, defineLoader, useAction, Form } from 'hono-preact';
import { renderPage, HonoContext } from 'hono-preact/server';
import { defineApp } from 'hono-preact/vite';
```

Workspace packages stay as build-time concerns. The published surface is one package, three subpaths. The user never types `@hono-preact/*`.

## 9. Cuts (things that get deleted)

Without users, cuts cost nothing. Aggressive list:

- `definePage` as a public wrapper (the route file's default export *is* the component; bindings come via `routes.ts` references).
- `useLoaderData<typeof loader>()` — replaced by `loader.useData()`.
- `cacheRegistry` (registry, `register`, `invalidate('name')`, `acquire`) — replaced by `loader.invalidate()`.
- `env.current` from the public surface — internal only.
- `ActionStub` as a public type — internal.
- `apps/app/src/iso.tsx` — no manual route table.
- `apps/app/src/client.tsx` — framework-provided.
- `apps/app/src/shims/process.ts` — already done (2026-05-09).
- The `@hono-preact/iso` / `@hono-preact/server` / `@hono-preact/vite` user-visible split.
- Vite alias block in `vite.config.ts` — implicit in `defineApp` for monorepo dev mode.
- `import.meta.env.PROD` ternary in `Layout` for the client script tag.
- Manual `<Router>` and `<Route>` use in user code — internal.

## 10. Out of scope for v0.1

To actually ship rather than design forever, hard lines:

- **No data layer.** Loaders return whatever you give them.
- **No auth primitive.** Guards are the surface; user wires the auth.
- **No deploy adapter abstraction.** Cloudflare Workers only at v0.1.
- **No middleware system beyond Hono's.** Anything middleware-shaped goes through the guards list or `app.use()` in `api.ts`.
- **No client-state library.** Loader data + optimistic actions are the data flow story.
- **No nested loaders / parallel data graph.** One loader per route, period. Compose inside the loader function if you need parallelism.
- **No layout-state preservation primitive.** Layout-owned local state (filter inputs, scroll position, badges, sidebar pinning) is lost when the layout unmounts on navigation away from its segment. v0.1 users handle this in userland (module-level singletons or `sessionStorage`). A framework primitive — working name `useSurvivingState(key, initial)`, a `useState` whose value survives the component's own remount within a session — would remove the boilerplate once 2–3 call sites surface. The demo's movies-layout filter and watched badge are the first call sites worth tracking.

Each is reasonable post-v0.1. None is the difference between "this framework is interesting" and "this framework is not."

## 11. Sequencing

The order matters because some pieces unlock others.

1. **`defineRoutes` + route-table-as-manifest in `@hono-preact/vite`.** Everything hangs off this. The plugin reads the route table at build time, emits the loader/action handler glue, produces a static manifest for the runtime. Migrate the demo app's `iso.tsx` + `server.tsx` glob calls to the new shape as the integration test.
2. **Methods on loader/action refs.** `loader.useData()`, `loader.invalidate()`, `action.useAction()`. Kill `cacheRegistry` and the `useLoaderData<typeof loader>()` pattern in the demo.
3. **`defineApp()`** in the vite plugin, absorbing the `vite.config.ts` body.
4. **Framework-provided client entry** + `<ClientScript />` + `<Head>`. Kills `client.tsx` and the script-tag branching.
5. **Streaming loaders + `<Form>` streaming parity.** The differentiator.
6. **Single guards list** + `ActionGuardError` status type fix.
7. **One package, three subpaths** consolidation. Last because it touches everything.
8. **Evaluate auth + cookies story.** The spec carves out "no auth primitive, guards are the surface" (section 10). Before the launch README claims that as a feature, validate that users can actually build the common patterns on top of what v0.1 ships: session cookies, JWT bearer, OAuth callback handling, signed/encrypted cookies, CSRF on `<Form>` actions, logout/invalidate. Inputs: Hono's cookie helpers, the guards list, loader/action context. Outputs: either (a) a docs recipe page + a working example flow in the demo app, or (b) an identified gap that genuinely needs a framework primitive (in which case it becomes its own sequenced item). Done before 10 so the launch README has a real answer to "how do I add login?".
9. **Audit Hono primitives + packages we bypassed.** Walk Hono's built-in helpers (`hono/cookie`, `hono/jwt`, `hono/jsx`, `hono/streaming`, `hono/html`, `hono/context-storage`, `hono/cors`, `hono/csrf`, `hono/secure-headers`, `hono/cache`, `hono/etag`, `hono/logger`, `hono/timing`, etc.) and the `@hono/*` ecosystem packages. For each piece we currently reimplement or sidestep in `packages/server`, `packages/iso`, or `packages/vite`, decide: keep our version (with a recorded reason), replace with the Hono one, or thinly wrap. Output is a markdown table in `docs/superpowers/research/` plus any follow-up refactor items spun out as their own work. Runs alongside item 8 since both feed the launch story.
10. **README, landing page, npm publish.** This is the moment, not an afterthought.

Items 1–4 are the architectural rewrite. Item 5 is the differentiator. Items 6–7 are the polish. Items 8–9 are the pre-launch evaluations. Item 10 is the launch.

## 12. The day-one user experience

If steps 1–7 land, the README's first code block is the actual code:

```ts
// vite.config.ts
import { defineApp } from 'hono-preact/vite';
export default defineApp();
```

```ts
// src/routes.ts
import { defineRoutes } from 'hono-preact';
export default defineRoutes([
  { path: '/', view: () => import('./views/home') },
]);
```

```tsx
// src/views/home.tsx
export default function Home() {
  return <h1>Hello</h1>;
}
```

```tsx
// src/Layout.tsx
import { ClientScript, Head } from 'hono-preact';
export default function Layout({ children }) {
  return (
    <html>
      <Head defaultTitle="hono-preact" />
      <body><main id="app">{children}</main><ClientScript /></body>
    </html>
  );
}
```

Four files. One framework opinion per file. That's the spec.
