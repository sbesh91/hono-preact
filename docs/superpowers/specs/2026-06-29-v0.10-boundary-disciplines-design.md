# v0.10.0 — Boundary disciplines (tRPC adoption, low-risk cut)

Status: approved design, pre-implementation
Date: 2026-06-29
Tracking: closes part of [#200](https://github.com/sbesh91/hono-preact/issues/200) (items #2, #3, #4, #6)
Published baseline: v0.9.0 (tag `fdd5423`)

## Summary

v0.10.0 adopts four tRPC-inspired *boundary disciplines* from the #200 epic,
chosen as the low-risk, additive cut: nothing breaks the public surface, and
three of the four mostly assemble parts that already exist. The two items from
#200 that carry real design tension or architectural reach (the pluggable data
transformer #1, and typed middleware context accumulation #8) are deliberately
deferred to a later release with their own design spikes.

In scope:

- **#3 Inference helper types** — public extractors over existing phantoms.
- **#2 Public server-side caller** — in-request loader/action composition with
  no HTTP round-trip.
- **#4 Typed error codes over `deny()`** — a named code vocabulary decorating
  the existing numeric status, including a status-inferring overload.
- **#6 Cancellation parity for actions/forms** — wire an `AbortController` into
  `mutate` and `<Form>` submit.

Explicitly out of scope for v0.10: #1 (transformer), #5 (output validation;
rides #1's return-boundary work), #7 (resumable SSE; independent track), #8
(typed middleware context; design spike first).

## Why these four, why now

Grouped by risk rather than the issue's value/effort tiers:

- #3 is a pure type addition; the `ActionRef.__phantom` tuple and `LoaderRef`'s
  `T` already encode everything (`packages/iso/src/action.ts:22`,
  `packages/iso/src/define-loader.ts:94-154`).
- #2's seams already exist behind `@hono-preact/iso/internal` (`dispatchServer`,
  `runRequestScope`) and the run path is proven in
  `packages/server/src/loaders-handler.ts:258`. We expose a sanctioned public
  caller built on them, scoped to the in-request case so we do not have to mint
  a synthetic Hono `Context`.
- #4 is an additive optional field on the deny outcome
  (`packages/iso/src/outcomes.ts:18-24`); the wire envelope already carries an
  optional `data` slot, so `code` rides the same path.
- #6 is contained client-side parity work; the server already honors
  `ctx.signal`, loaders already thread an `AbortSignal`
  (`packages/iso/src/internal/use-loader-runner.tsx:95`).

No public breaking change. Pre-1.0, but this release does not need a break.

## Item #3 — Inference helper types

### Surface

New public type exports from `packages/iso/src/index.ts`:

```ts
export type InferActionPayload<A> = A extends ActionRef<infer P, any, any> ? P : never;
export type InferActionResult<A>  = A extends ActionRef<any, infer R, any> ? R : never;
export type InferActionChunk<A>   = A extends ActionRef<any, any, infer C> ? C : never;
export type InferLoaderData<L>    = L extends LoaderRef<infer T, any> ? T : never;
```

### Decisions

- **Naming is prefixed** (`InferAction*` / `InferLoader*`), not the issue's bare
  `InferPayload` / `InferResult` / `InferData`. A bare `InferResult` is
  ambiguous across the two ref kinds; the prefix removes the ambiguity at the
  call site.
- **They resolve to the authored `T`** (the server-side type), not
  `Serialize<T>`. `Serialize` is already public, so a consumer who needs the
  wire shape composes `Serialize<InferActionResult<typeof action>>`. One source
  of truth; no duplicate "wire vs server" extractor pair.
- The `LoaderRef` second type parameter (`Live`) is matched with `any` so both
  live and non-live loaders are accepted.

### Where it lives

A small new module `packages/iso/src/infer.ts` (keeps `action.ts` /
`define-loader.ts` focused), re-exported from `index.ts`. The extractors are
type-only; no runtime.

### Tests

`packages/iso/src/__tests__/infer.test-d.ts` (vitest `test:types` mode):

- `InferActionResult<typeof someAction>` equals the authored result type.
- `InferActionPayload` equals the schema's `InferOutput` when `input` is set,
  and the explicit `TPayload` otherwise.
- `InferLoaderData<typeof someLoader>` equals the authored `T` (NOT
  `Serialize<T>`); assert a `Date` field stays `Date`.
- Composition: `Serialize<InferActionResult<...>>` degrades the `Date` to
  `string`, proving the building-block story.

### Risk

Lowest in the release. Type-only, additive.

## Item #2 — Public server-side caller (in-request composition)

### Surface

A `call` method on the server ctx that loaders and actions already receive,
letting a handler invoke another loader/action without an HTTP round-trip,
reusing the current request scope and Hono `Context`:

```ts
// inside a loader or action body, on the ctx you already hold
const movie  = await ctx.call(movieLoader, { location });   // loader from loader (SSR)
const result = await ctx.call(chargeAction, payload);       // action from action
```

Plus a test-time factory that binds a caller to a user-provided Hono `Context`:

```ts
import { createCaller } from '@hono-preact/iso';
const caller = createCaller(c); // c: a Hono Context the test constructs
const data = await caller.call(movieLoader, { location });
```

### Semantics

- **Reuses the live request scope.** The caller detects an active request scope
  via `getRequestStore()`. When one is active (called from inside a handler), it
  runs the inner procedure directly in that scope, so the Hono context and
  request-scoped ALS state are inherited and the per-request ALS context is not
  re-entered. `createCaller(c)` used standalone (a test, no active scope) opens a
  fresh `runRequestScope(fn, { honoContext: c })` per call. Note: the caller
  invokes `ref.fn` directly, so it does NOT consult or populate the loader's
  result cache (`ref.cache`); that memoization lives only in the SSR page-render
  path (`use-loader-runner`), and the HTTP loader handler likewise calls the
  loader function directly. A `ctx.call(loaderB)` therefore re-runs loaderB even
  if the page render also ran it (always-correct data, possibly redundant work);
  result-cache sharing is a deliberate non-goal for this release.
- **Runs validation.** Input is validated through the ref's
  `searchSchema` / `paramsSchema` (loader, via `coerceLoaderLocation`) or
  `input` (action, via `validateWithSchema`) exactly as the HTTP handlers do, so
  a caller cannot bypass the schema. A validation failure surfaces as the same
  typed deny outcome the HTTP path produces (see #4), returned as a short-circuit
  `CallResult`, not a raw throw.
- **Runs the ref's OWN per-unit middleware, not the app/page chain.** The caller
  dispatches through `dispatchServer` with `partitionUse(ref.use)`, i.e. only the
  procedure's own `use` middleware. It deliberately does NOT re-run the app-level
  (`appConfig.use`) or page-level middleware, because those already ran for the
  outer request; re-running them would double-fire request-level guards
  (auth, logging). Honest composition: the inner procedure's own guards apply;
  the request-level guards are not re-applied.
- **Returns raw `T` / `TResult`.** No JSON round-trip, so the return type is the
  authored type, not `Serialize<T>`. This is the deliberate difference from the
  client hooks: server-side composition keeps `Date` as `Date`.
- **Outcome shape (pinned).** The caller returns a discriminated
  `CallResult<T>`:

  ```ts
  export type CallResult<T> =
    | { ok: true; value: T }
    | { ok: false; outcome: Outcome }; // redirect | deny | timeout | render
  ```

  A `redirect` / `deny` from the inner procedure (or its middleware / schema
  validation) is returned as `{ ok: false, outcome }`, NOT thrown as a
  navigation. Callers narrow on `ok` and decide what to do (re-throw to
  propagate, inspect `outcome.code`, etc.).

### Decisions

- **Scope is in-request composition only.** The "unit-test a handler with zero
  request context" case needs a synthetic Hono `Context`. We do NOT mint one in
  v0.10. `createCaller(c)` accepts a `Context` the test brings (Hono makes this
  trivial via `app.request` / a constructed context), which covers the testing
  unlock without us owning context fabrication. Full synthetic-context minting
  is deferred.
- **No middleware bypass** in v0.10. If a real need appears, add an opt-out
  later; defaulting to "runs middleware" is the safe, honest default.
- **Streaming loaders/actions:** v0.10 `call` supports the single-value path.
  Calling a streaming/live procedure returns its terminal value where one
  exists; the streaming-consumption ergonomics are deferred (note it explicitly
  in docs and throw a clear error for the live-loader case, which has no single
  value).

### Where it lives

- New module `packages/iso/src/server-caller.ts` holding the caller core
  (validation + `runRequestScope` reuse + `dispatchServer`), factored so both
  `ctx.call` and `createCaller` share one implementation.
- `LoaderCtx` (`define-loader.ts`) and `ActionCtx` (`action.ts`) gain a `call`
  method in their type, populated where the server handlers construct the ctx
  (`loaders-handler.ts`, `page-actions-handler.ts`).
- `createCaller` exported from `index.ts`.
- The caller imports the existing internal seams directly (it lives in `iso`,
  same package), so nothing new is added to the `/internal` door.

### Tests

- Unit: `caller.call(loader, { location })` returns the authored type, coerces
  valid params (success path), runs the schema (rejects bad input -> deny
  outcome), runs a middleware guard (a `deny` from middleware short-circuits to
  an outcome result), and returns an action's value on success.
- Composition (loader-calls-loader via `ctx.call`) is exercised in the Task 6
  `ctx.call` work, where the in-request scope path is wired end-to-end.
- Type test: return type is `T`, not `Serialize<T>` (asserted via test 1's
  `Date`-stays-`Date` check).

### Risk

The only item with design surface; the narrowing to in-request composition plus
a bring-your-own-context test factory keeps it bounded. The run path is a
re-composition of code already exercised by the HTTP handlers.

## Item #4 — Typed error codes over `deny()`

### Surface

A named code vocabulary decorating the existing numeric status:

```ts
export type DenyCode =
  | 'BAD_REQUEST'        // 400
  | 'UNAUTHORIZED'       // 401
  | 'FORBIDDEN'          // 403
  | 'NOT_FOUND'          // 404
  | 'CONFLICT'           // 409
  | 'UNPROCESSABLE'      // 422
  | 'TOO_MANY_REQUESTS'  // 429
  | 'INTERNAL';          // 500
```

`DenyOutcome` and `DenyInput` gain an optional `code?: DenyCode`
(`packages/iso/src/outcomes.ts`). `status` stays authoritative; `code` is
optional decoration.

### The status-inferring overload (included)

```ts
deny('NOT_FOUND')                       // -> status 404, code 'NOT_FOUND'
deny('FORBIDDEN', 'Members only')       // -> status 403, code 'FORBIDDEN', message
deny({ code: 'CONFLICT', data })        // -> status 409, code 'CONFLICT', data
deny(409, 'Already exists')             // unchanged: numeric status, no code
deny({ status: 409, code: 'CONFLICT' }) // explicit both; status wins if they disagree
```

A `DENY_CODE_STATUS` map (`code -> ErrorStatusCode`) provides the default
status when a code is given without an explicit status. When both are present
the explicit `status` is authoritative (no throw on mismatch; document the
precedence).

### Wire + client threading

`code` must survive to the client and be readable without string-sniffing:

- `serializeActionOutcome` / the deny envelope carry `code`
  (`packages/iso/src/internal/action-envelope.ts`).
- `StoredActionResult` deny variant gains `code`
  (`packages/iso/src/internal/action-result-store.ts`).
- The client `deny` callback in `useAction`'s `applyDecodedOutcome`
  (`action.ts:486`) receives `code`; it flows into `useActionResult` /
  `ActionResult` so a component can `switch (result.code)`.
- The SSR no-JS deny re-render path also carries `code`: the `Outcome` stored
  in the `ActionResultSlot` already holds it once `DenyOutcome` gains `code`,
  and `buildActionResultContext` (`packages/server/src/render.tsx:52`) +
  `ActionResultContextValue` (`action-result-context.tsx`) thread it to the
  same `useActionResult` reader.

**Scoped to the action path.** The loader deny -> client path currently
surfaces an unstructured `Error` (the loader error arm is `Error | null`; there
is no structured `deny` result variant like the action path's `ActionResult`).
Threading a typed `code` to a loader would require introducing a new structured
loader-error surface (e.g. a `DenyError` carrying `status` + `code`), which is
its own public-API decision. v0.10 carries `code` on the action path only;
loader-deny-code is deferred to a follow-up. We do NOT emit `code` on the loader
deny envelope until the client surfaces it (no dead wire bytes).

### Decisions

- **Scope to `deny()`.** A generic non-deny throw still degrades to
  `{ __outcome: 'error', message }` with no code (unchanged). #4 does not try to
  code-tag arbitrary thrown errors.
- **Closed string union, not an open enum.** A fixed union is `switch`-able with
  exhaustiveness and adds zero runtime. Custom/app-specific codes are out of
  scope; the existing `data` slot already carries app-specific payloads.
- **Default messages** keep the existing status-aware fallback in `deny()`.

### Tests

- `deny('NOT_FOUND')` produces `{ status: 404, code: 'NOT_FOUND' }`.
- Explicit status overrides the code's default.
- Round-trip: a server `deny('FORBIDDEN')` surfaces `result.code === 'FORBIDDEN'`
  on the client (`useActionResult`), and on the loader deny path.
- Type test: `result.code` is `DenyCode | undefined` and narrows in a `switch`.
- Back-compat: `deny(403, 'msg')` still yields `code: undefined`.

### Risk

Additive optional field; non-breaking. Surface area is the wire/store/client
threading, which is mechanical but must be done end-to-end (a code that stops at
the envelope is a silent half-feature).

## Item #6 — Cancellation parity for actions/forms

### Surface

- `useAction().mutate` accepts an optional caller signal:
  `mutate(payload, { signal })`.
- Both `mutate` and `<Form>` submit abort their in-flight `fetch` on unmount.

### Semantics

- **Unmount aborts the in-flight request.** A per-call `AbortController` is
  created in `mutate` / submit; an effect cleanup (or a ref-tracked controller)
  aborts it on unmount. The `fetch` calls (`action.ts:407` and `:413`,
  `form.tsx:277`) receive `controller.signal`. If a caller passes
  `{ signal }`, it is linked so either source aborts the request.
- **Do NOT auto-abort a prior in-flight mutation on a new `mutate`.** Loaders
  abort-previous, but concurrent mutations are frequently intentional (e.g. a
  row of independent saves). Preserve current fire-many semantics. A caller that
  wants abort-previous brings its own `signal`.
- **AbortError is classified, not recorded as an error outcome.** An
  unmount-cancel must not `setState` after unmount and must not write a generic
  `kind: 'error'` to the action-result store. Detect `AbortError` in the catch
  and short-circuit: no `setError`, no `recordOutcome`, return a quiet
  cancelled result (e.g. `{ ok: false, error }` where `error` is the AbortError,
  but guard the state writes).

### Decisions

- Mirror the loader cancellation model only where it makes sense: parity on
  *unmount* cancellation and *opt-in caller* cancellation, not on
  *abort-previous* (the one place mutations legitimately differ from loaders).
- `<Form>` native no-JS POST path is unaffected (there is no JS to cancel);
  cancellation applies to the JS-enhanced submit only. Note in docs.

### Tests

- Unmounting a component mid-`mutate` aborts the underlying request and does not
  throw a "setState after unmount" warning or record an error outcome.
- A caller `signal` aborts an in-flight `mutate`.
- Two concurrent `mutate` calls both complete (no implicit abort-previous).
- `<Form>` submit aborts on unmount.

### Risk

Contained, non-breaking. The subtlety is AbortError classification (don't let a
cancel look like a failure).

## Cross-cutting

### Breaking changes

None expected. #3 adds types, #2 adds `ctx.call` + `createCaller`, #4 adds an
optional `code` and a new `deny()` overload, #6 adds an optional `mutate` arg
and internal abort wiring. Existing call sites compile and behave unchanged.

### Sequencing

1. **#3 inference helpers** — isolated, type-only; lands first.
2. **#4 typed error codes** — touches the deny wire/store/client path; lands
   before #2 so the server caller can reuse the typed deny outcome.
3. **#6 cancellation** — independent client-side work; lands any time after #3.
4. **#2 server caller** — last; the only item with design surface, and it
   benefits from #4's typed deny outcomes being in place.

### Docs

Per the project docs rules (no migration breadcrumbs; document what *is*; CSS/
Tailwind parity where applicable; self-contained API tables):

- Inference helpers: a reference section with the four extractors and the
  `Serialize<>` composition example.
- Server caller: a guide page (loader-composing-loader, action-calling-action,
  the `createCaller(c)` test pattern, the raw-`T`-not-`Serialize<T>` note, and
  the streaming/live-loader limitation).
- Error codes: a `DenyCode -> status` table and a client `switch (result.code)`
  example.
- Cancellation: a note on unmount cancellation, the opt-in `signal`, the
  no-abort-previous semantics, and the no-JS Form caveat.

### Verification (pre-push, per CLAUDE.md)

Each item is verified with `pnpm typecheck` (shared-type changes are not caught
by `test:types` alone), plus the full eight-step CI mirror before any PR. #2 and

### Non-goals (this release)

  needs its own spec; #8 in particular is a design spike (middleware signature
  change).
- Central `AppRouter` / single typed client, request batching — permanent
  non-goals per #200; the colocation + build-transform model is the
  differentiator.
