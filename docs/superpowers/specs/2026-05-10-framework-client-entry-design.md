# Framework-provided Client Entry, `<ClientScript />`, `<Head>`, `<ViewTransitions />`, `useRouteChange`

**Date:** 2026-05-10
**Status:** Draft
**Implements:** Â§4 of `2026-05-09-v0.1-framework-direction.md` (with extensions for view-transition handling that the original spec did not address).

## TL;DR

`apps/app/src/client.tsx` and `apps/app/src/iso.tsx` go away. The framework owns the client entry as a virtual module (`virtual:hono-preact/client`) generated by `@hono-preact/vite`. The user's `Layout.tsx` becomes pure chrome that receives `children`, with three new framework components dropped in: `<Head>` for the document head, `<ClientScript />` for the dev/prod-aware `<script>` tag, and `<ViewTransitions />` to opt into View Transitions on route change. A `useRouteChange((to, from) => void)` hook ships as the escape hatch for analytics, scroll restoration, or custom transitions.

The day-after-item-4 user code:

```tsx
// apps/app/src/Layout.tsx
import { ClientScript, Head, ViewTransitions } from 'hono-preact';
import root from '@/styles/root.css?url';

export default function Layout({ children }) {
  return (
    <html>
      <Head defaultTitle="hono-preact">
        <link rel="stylesheet" href={root} />
      </Head>
      <body class="bg-gray-300 isolate">
        <main id="app">{children}</main>
        <ClientScript />
        <ViewTransitions />
      </body>
    </html>
  );
}
```

That is the entire client-side glue the user writes. No `hydrate`, no `LocationProvider`, no `<Routes>`, no script-tag ternary, no `<HonoContext.Provider>`, no manual charset/viewport, no `import.meta.env.PROD` branch. All gone.

## Why this shape

Item 3 already moved the server entry into the framework. Item 4 does the same for the client side and finishes the "Layout is just chrome" promise from Â§1 of the v0.1 framework-direction spec.

Two places where the design diverges from a pure read of Â§4 of that spec, both validated during brainstorming:

- **Layout receives `children` only**, not `{ children, context }`. The simpler signature matches the spec's pitch ("Layout is just chrome"). Users who really need request data peek through a loader, not through Layout props. `HonoContext`/`useHonoContext` stay exported as advanced primitives, but the framework-generated wrap no longer includes the provider in the demo's default path.
- **View transitions are opt-in via `<ViewTransitions />`**, not auto-enabled. Today's `iso.tsx` hard-codes `document.startViewTransition(() => flushSync(() => {}))` on every route change. After this PR, the user no longer writes that file, so the behavior has to come from somewhere visible in the user's source. A drop-in component beats a hidden default. `useRouteChange((to, from) => void)` is the escape hatch for everything else (analytics page-views, scroll restoration, custom segment-scoped transitions).

## Public API

Three new components and one hook, all exported from `hono-preact` (and `@hono-preact/iso` underneath, until item 7's consolidation).

### `<ClientScript />`

```ts
declare function ClientScript(): VNode;
```

Renders `<script type="module" src="...">`. The `src` resolves to:
- **Prod:** `/static/client.js` (the built client bundle).
- **Dev:** Vite's URL for the `virtual:hono-preact/client` module (typically `/@id/__x00__virtual:hono-preact/client`).

No props for v0.1. CSP nonces and similar refinements wait for a real user request.

### `<Head>`

```ts
declare function Head(props: {
  defaultTitle?: string;
  children?: ComponentChildren;
}): VNode;
```

Renders the document `<head>` element with built-in defaults:
- `<meta charset="utf-8">`
- `<meta name="viewport" content="width=device-width,initial-scale=1.0">`
- `<title>{defaultTitle ?? ''}</title>` (used as fallback when no `useTitle` from a page sets one)

Children are emitted inside `<head>` for user static tags (stylesheet links, favicons, custom meta). Hoofd-collected dynamic tags (per-route `useTitle`/`useMeta`/`useLink`) continue to be appended before `</head>` by `renderPage`'s existing post-process. The dynamic title overrides the default; this is hoofd's normal behavior.

### `<ViewTransitions />`

```ts
declare function ViewTransitions(): VNode;
```

Renders `null`. On mount (client-only), registers a route-change subscriber that wraps the next preact commit in `document.startViewTransition(() => flushSync(() => {}))`. Feature-detects `document.startViewTransition`; falls back to a regular commit when missing. Unsubscribes on unmount. SSR no-op.

Multiple `<ViewTransitions />` instances dedupe via a module-level counter (the wrap fires once per route change as long as at least one is mounted, regardless of count).

For per-segment scoping (e.g., "only animate within `/movies/*`"), use `useRouteChange` instead. Pattern-matching is intentionally not built into `<ViewTransitions />` at v0.1; it's a non-breaking addition if real demand surfaces.

### `useRouteChange`

```ts
declare function useRouteChange(
  handler: (to: string, from: string | undefined) => void
): void;
```

Subscribes the handler on mount, unsubscribes on unmount. Server-side: no-op. Client-side: fires on every route change with the new path and the previous path (`undefined` on the first navigation after hydration).

Use cases (the escape hatch covers all of these):

- Analytics page-views: `useRouteChange((to) => analytics.page(to))`.
- Scroll restoration: `useRouteChange((to, from) => { if (from) window.scrollTo(0, 0); })`.
- Segment-scoped view transitions: `useRouteChange((to, from) => { if (to.startsWith('/movies/') && from?.startsWith('/movies/')) document.startViewTransition(() => flushSync(() => {})); })`.

## Internals

### Generated client entry

`@hono-preact/vite` registers `virtual:hono-preact/client`. Implemented via `resolveId`/`load` hooks (the canonical pattern is in `client-shim.ts`; the server-entry pivot to disk in item 3 was forced by `@hono/vite-build`'s glob constraint, which doesn't apply on the client side).

Generated source (effectively):

```tsx
// virtual:hono-preact/client (generated)
import { hydrate } from 'preact';
import { LocationProvider } from 'preact-iso';
import { Routes } from '@hono-preact/iso';
import { __dispatchRouteChange } from '@hono-preact/iso/internal';
import routes from '/abs/path/to/src/routes.ts';

let lastPath: string | undefined;

hydrate(
  <LocationProvider>
    <Routes
      routes={routes}
      onRouteChange={(path) => {
        const from = lastPath;
        lastPath = path;
        __dispatchRouteChange(path, from);
      }}
    />
  </LocationProvider>,
  document.getElementById('app') as HTMLElement
);
```

The `clientShimPlugin` is updated to inject the `process` shim into this virtual entry alongside the user's `clientEntry` (when one exists; the framework's default flow no longer has a user-authored `clientEntry`).

### Generated server entry shape change

The server entry's catch-all currently reads:

```tsx
.get('*', (c) => renderPage(c, h(Layout, { context: c })))
```

After item 4:

```tsx
.get('*', (c) =>
  renderPage(c, h(Layout, null,
    h(LocationProvider, null, h(Routes, { routes }))
  ))
)
```

`Layout` receives `children` (the LocationProvider + Routes tree). The `defaultTitle` argument to `renderPage` is dropped from the framework's invocation; `<Head defaultTitle="...">` carries it forward via hoofd. `renderPage`'s signature retains `options.defaultTitle` for backward compatibility (advanced users with custom server entries can still pass it), but the framework's generated path no longer uses it.

### Internal subscriber registry

Lives in `@hono-preact/iso/internal/route-change.ts` (private):

```ts
type Sub = (to: string, from: string | undefined) => void;
const subs = new Set<Sub>();

export function __dispatchRouteChange(to: string, from: string | undefined): void {
  for (const cb of subs) cb(to, from);
  if (viewTransitionEnabled > 0 && typeof document !== 'undefined' && document.startViewTransition) {
    document.startViewTransition(() => flushSync(() => {}));
  }
}

export function __subscribeRouteChange(sub: Sub): () => void {
  subs.add(sub);
  return () => { subs.delete(sub); };
}

let viewTransitionEnabled = 0;
export function __enableViewTransitions(): () => void {
  viewTransitionEnabled++;
  return () => { viewTransitionEnabled--; };
}
```

`<ViewTransitions />` calls `__enableViewTransitions()` in its mount effect. `useRouteChange` calls `__subscribeRouteChange()`. Both unsubscribe on unmount. Server-side: the effects don't run, the registries stay empty, and `__dispatchRouteChange` is never called (the generated client entry only registers itself in the browser).

The double prefix (`__`) signals "internal" without making the names runtime-private; future bundling can mangle them away.

### `<ClientScript />` resolution

Inside the component, `import.meta.env.PROD` is the discriminator (Vite replaces it at build time):

```tsx
export function ClientScript() {
  const src = import.meta.env.PROD
    ? '/static/client.js'
    : '/@id/__x00__virtual:hono-preact/client';
  return <script type="module" src={src} />;
}
```

Same trick today's hand-written Layout uses; the framework now owns it.

The dev URL convention (`/@id/__x00__virtual:...`) is Vite's documented way to fetch a virtual module from the dev server. If Vite changes this in a future major (unlikely), the framework swaps the literal here and users get the fix transparently.

## Cuts

What this PR deletes or simplifies:

- **`apps/app/src/client.tsx`** â€” gone. Generated virtual entry takes over.
- **`apps/app/src/iso.tsx`** â€” gone. The `Base` component (LocationProvider + Routes wrap with view transitions) moves into the framework's generated client entry. View transitions become opt-in via `<ViewTransitions />` in the user's Layout.
- **The `import.meta.env.PROD ? ... : ...` ternary in `apps/app/src/Layout.tsx`** â€” gone. `<ClientScript />` owns the branching.
- **The `<HonoContext.Provider value={props}>` wrap in `apps/app/src/Layout.tsx`** â€” gone. No consumers in the demo. `HonoContext`/`useHonoContext` stay exported for advanced users; they can re-add the provider in a custom server entry if they need it.
- **The manual `<head>` block in `apps/app/src/Layout.tsx`** (charset, viewport, stylesheet link). `<Head>` provides charset and viewport as defaults; the stylesheet link goes inside `<Head>` as a child.
- **The `LocationProvider` import in `apps/app/src/Layout.tsx`** â€” gone. Framework owns it.
- **The `Base` import in `apps/app/src/Layout.tsx`** â€” gone with iso.tsx.

After: Layout is the four-import, ~12-line file shown in the TL;DR.

## Behavior parity to verify

| Today | After item 4 |
|---|---|
| `/`, `/movies`, `/movies/:id`, `/watched` SSR-render with full HTML | Same routes, same SSR output. Head includes charset, viewport, title, plus hoofd-collected per-route tags. |
| Hydration: client takes over from SSR HTML | `<ClientScript />` resolves to a working URL in dev and prod; hydration mounts at `#app`. |
| View transitions on route change in supported browsers | `<ViewTransitions />` mounted in Layout. Chrome animates, browsers without `document.startViewTransition` fall back to a normal commit. |
| `useTitle('Movies')` from a page sets `<title>` | Hoofd hooks still work. `renderPage`'s post-process unchanged. |
| Stylesheet link loads | `<link rel="stylesheet" href={root} />` inside `<Head>` works because `<Head>` renders children. |
| `useHonoContext()` in a custom layout returns the Hono context | Still works in custom server entries. The framework's default-generated server entry no longer wraps `<HonoContext.Provider>`, so `useHonoContext` is `undefined` in the demo. Documented as such. |
