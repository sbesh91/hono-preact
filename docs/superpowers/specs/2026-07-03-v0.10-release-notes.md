# hono-preact v0.10.0

The headline of v0.10 is the **loader inversion**: route-binding becomes a
composition (`serverRoute(r).loader`), bare `defineLoader` becomes the
route-independent primitive, and streaming is expressed as a loader rather than a
separate primitive. Several smaller boundary disciplines ship alongside it.

## Breaking changes

### 1. Loader inversion: route-binding by composition

Server loaders are now bound to a route by composition instead of by colocation
alone:

- **`serverRoute('/items/:id').loader(fn)`**: route-bound. The ctx carries a
  typed `location` (`pathParams`/`searchParams`), the loader inherits the route
  node's `use` chain (the page/auth tier), and it participates in SSR.
- **`defineLoader(fn)`**: route-independent. The ctx is `{ c, signal, call }`
  with **no `location`**, the loader runs with the app-level `use` chain plus its
  own per-loader `use` (no page/route tier), and it is consumable anywhere
  (including outside the router).

**Migration:**

- **`LoaderCtx` has no `location` by default.** A bare `LoaderCtx` (no generic)
  is the standalone shape and does not expose `location`. A loader that reads
  `ctx.location` must be route-bound: author it with `serverRoute(r).loader(fn)`
  (which infers the located ctx) or annotate the ctx explicitly as
  `LoaderCtx<RouteParams<'/items/:id'>>`. A bare `defineLoader` that reads
  `ctx.location` no longer type-checks.
- **`defineLoader` no longer takes a route id.** The
  `defineLoader('/movies/:id', fn, opts)` route-string overload is removed;
  `defineLoader(fn, opts?)` is now always the route-independent (standalone)
  constructor. Author route-bound loaders through
  `serverRoute('/movies/:id').loader(fn, opts)` exclusively.
- **A bare `defineLoader(fn, opts)` no longer accepts `paramsSchema`,
  `searchSchema`, or `params`** (a standalone loader has no `location` to
  validate). Move param/search validation onto the route-bound form:
  `serverRoute(r).loader(fn, { paramsSchema, searchSchema })`.
- **Route auth guards come from route binding.** A `serverRoute(r).loader` is
  server-authoritative: its route-node `use` guards resolve from its own declared
  route pattern (`byPattern(__routeId)`). A bare `defineLoader` is
  route-independent and runs with the app-level `use` chain plus its own
  per-loader `use` (no page/route tier), by design (the "composition-evident
  auth" model). When a loader needs a route-level guard, bind it with
  `serverRoute(r).loader`, or put the guard on the unit via a per-loader `use`.
  (The v0.9 loaders RPC resolved its page tier from the client-sent
  `location.path`, so it was never a server-authoritative gate for a bare loader;
  v0.10 does not remove a real boundary, it makes the contract explicit.)
- **`serverRoute(r).liveLoader(...)` is removed and `LiveLoaderOptions` is no
  longer exported.** A route-bound live subscription is now composed from
  `liveStream`: `serverRoute(r).loader(liveStream({ topic, load }))`. The
  `cache` / `use` / `timeoutMs` that `LiveLoaderOptions` carried move into the
  loader's options: `serverRoute(r).loader(liveStream({ topic, load }), { cache, use, timeoutMs })`.
- **A streaming loader's form is now driven by its function's return type, not a
  `{ live }` flag.** A loader fn that returns an `AsyncGenerator` is a streaming
  loader (accumulating `.View(render, { initial, reduce })` only); one that
  returns a `Promise` is single-value (`.View(render)` plus `.Boundary` and
  `useData()`). Pass `{ live: true }` in the options to opt a hand-written
  streaming loader out of SSR (a client-only subscription that never runs during
  `renderToStringAsync`); `liveStream` sets the flag for you. One behavior shift
  to note: a finite, non-`live` streaming loader now defaults to no timeout,
  whereas a single-value loader still gets the handler's 30s default.

### 2. `LoaderState` / `StreamState` cold arm carries `data?: never`

The cold `loading` arm of `LoaderState<T>` and the cold `connecting` arm of
`StreamState<T>` now declare `data?: never` (instead of omitting `data`). This
makes `data` uniformly readable as `T | undefined` on the bare union without a
narrow, but it **breaks `in`-based narrowing**. Because the cold arm now declares
an optional `data`, TypeScript no longer excludes it from the `'data' in state`
true branch, so `if ('data' in state)` no longer narrows to a data-bearing arm
(`state.data` is `T | undefined` there). This is a type-level change only: at
runtime the cold arm object still carries no `data` key, so `'data' in state`
still evaluates to `false` for it. That type/runtime mismatch is why
`'data' in state` is no longer a safe discriminant.

**Migration:** narrow on `status` (`state.status === 'success'`), or use a
structural data-bearing check, instead of `'data' in state`. Read `data` only
after narrowing to a data-bearing status.

### 3. Realtime cross-runtime trap enforcement

- **`socket.data` / `conn.data` are now deeply `Readonly`.** On a hibernatable
  Cloudflare Durable Object an in-place mutation of the per-connection bag
  silently vanishes (it is JSON-serialized onto a forward header and re-read
  fresh per event); the read-only type turns that into a compile error at every
  nesting level, so a nested `conn.data.profile.name = ...` is rejected too, not
  just a top-level reassignment (a shallow `Readonly` froze only the top level).
  Migration: capture a closure variable in `open()` (sockets) or `onJoin()`
  (rooms) for mutable Node-only state, or use `setPresence` (rooms) for state
  that must evolve and broadcast.
- **A factory-less room's `conn.data` is now `undefined` (was `{}`)**, matching
  `socket.data` and the Cloudflare resolution path; the `defineRoom` `Data`
  generic defaults to `undefined`. Migration: add a `data` factory or guard the
  access.
- Not breaking: the Node dev server now warns when a `data` factory result
  exceeds the 6KB forward-header budget that throws at connect time on
  Cloudflare.

### 4. Hold-alive guarded navigation (advanced-consumer contract)

Behavior change, not a public API change: on a client navigation between routes
that have page-layer middleware (`use`), the previous route now stays visible
while the next route's middleware chain resolves, instead of blanking out. Apps
using the normal `Routes` mounter need no migration. The one contract change for
advanced consumers: the page-middleware host now relies on the surrounding
`Router` (provided by `Routes`) as its suspense boundary rather than carrying its
own, so code that mounts the host directly via `hono-preact/internal` without a
`Router` ancestor must add one.

### 5. `Envelope` (internal) `as` restricted to intrinsic elements

Advanced-consumer contract only: `Envelope` is exported from
`hono-preact/internal` (the loader-internal escape hatch). Its optional `as` prop
no longer accepts a custom component wrapper (`ComponentType`); it is restricted
to `keyof JSX.IntrinsicElements` (default `'section'`). A component wrapper could
not forward the ref the hydration de-orphan effect needs to identify its own DOM
node, so that path silently no-op'd the cleanup; removing it keeps the effect
always correct. Nothing in-repo used it. Consumers that need a custom wrapper
element should use `Page`'s `Wrapper` prop, the supported custom-wrapper knob.

## New (additive)

- **Server-side caller.** `createCaller(c)` builds an HTTP-free caller bound to a
  Hono `Context` for tests and server code; `ctx.call(ref, ...)` reaches a
  loader/action in-process, running the callee's own `use` (not the app or route
  tiers), returning the authored type (no `Serialize` round-trip). Public types:
  `ServerCaller`, `CallResult<T>`, `CallLoaderLocation`.
- **Route-bound server actions.** `serverRoute('/items/:id').action(fn, { input })`
  is the action analog of a route-bound loader: it resolves the action's
  page-tier `use` (auth) chain from its exact declared route pattern instead of
  fuzzy-matching the POST URL, closing the `/a/:x` vs `/a/:y` collision window a
  bare `defineAction` is subject to. The overloads mirror `defineAction` (pass
  `{ input }` to validate and infer the payload); unlike `.loader` there is no
  `ctx.location`. Consume it with `useAction(serverActions.x)`. A boot-time guard
  fails closed if a route-bound loader or action is mounted on a route other than
  the one it declares.
- **Typed deny codes.** `deny(code)` accepts a named `DenyCode`
  (`BAD_REQUEST`, `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `CONFLICT`,
  `UNPROCESSABLE`, `TOO_MANY_REQUESTS`, `INTERNAL`) that decorates the numeric
  status, so clients can `switch` on a typed code instead of sniffing the message.
  `status` stays authoritative. New exports: `DenyCode`, `DENY_CODE_STATUS`.
- **`liveStream({ topic, load })`.** Compose a route-independent live loader:
  `route.loader(liveStream({ topic, load }))` or a bare
  `defineLoader(liveStream(...))`.
- **Inference helpers.** `InferActionPayload`, `InferActionResult`,
  `InferActionChunk`, `InferLoaderData` over action/loader refs.
- **Opt out of the nav view transition (#165).** `skipNextNavTransition()`, the
  `<NavLink transition={false}>` prop, and `useNavigate`'s `{ transition: false }`
  option suppress the view transition for the next navigation.
- **Client-side schema pre-validation for `useAction` (#154).** When a `schema`
  is supplied, `useAction` validates the payload on the client before dispatch;
  the rejection shape matches the server's deny. The server still validates, so
  this is an ergonomic short-circuit, not a security boundary.
- **Colocated server-module auto-discovery + `src/server` registry
  (#215/#216/#217).** A `.server.*` file next to a route's inline
  `view`/`layout` import is discovered and wired automatically, so an explicit
  `server:` route field becomes an advanced override (non-sibling module,
  `server: false` opt-out, or runtime-generated route tables). Route-less server
  modules can live under `src/server/` at any depth.
