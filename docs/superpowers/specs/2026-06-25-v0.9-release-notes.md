# v0.9.0 Release Notes

`hono-preact` and `create-hono-preact` ship together at 0.9.0 (lockstep). `hono-preact-ui` is unchanged at 0.2.0.

**Design spec:** [2026-06-25-loader-loading-state-design.md](./2026-06-25-loader-loading-state-design.md)

---

## Breaking Changes

### 1. `.View()` / `useData()` now hand you a `LoaderState` / `StreamState` discriminated union; `reload` moves to `useReload()`; `fallback` removed

**Affects:** All call sites of `loader.View(render, opts)`, `loader.useData()`, and `loader.Boundary`.

**What changed:**

- The `.View(render)` render function and `loader.useData()` now return a discriminated union you pattern-match on `status`, instead of flat `{ data, loading, error, reload }` fields. Single-value loaders surface a `LoaderState<T>`; `live` loaders surface a `StreamState<T>`.
  - `LoaderState<T>` = `{ status: 'loading' }` | `{ status: 'success'; data: T }` | `{ status: 'revalidating'; data: T }` | `{ status: 'error'; error: Error; data: T }`.
  - `StreamState<T>` = `{ status: 'connecting' }` | `{ status: 'open'; data: T }` | `{ status: 'closed'; data: T }` | `{ status: 'error'; error: Error; data: T }`.
- `data` is **narrowed to `T`** in every value arm (`success` / `revalidating` / `error` for `LoaderState`; `open` / `closed` / `error` for `StreamState`). There is no more `T | undefined` to guard with `!`: the no-data arm (`loading` / `connecting`) simply has no `data` field. A cold error (no prior data) is caught by the loader's error boundary and the page `errorFallback` before the render function runs, so the `error` arm always carries `data`.
- `reload` is **no longer a render arg**. Read it from `useReload()` (`const { reload, reloading } = useReload()`), called at the top of the render function or in any descendant inside the loader boundary. `reloading` is `true` only during an explicit reload or revalidation, never the cold initial load.
- The `fallback` option on `.View(render, opts)` and the `fallback` prop on `loader.Boundary` are **removed**. Loading UI is the caller's responsibility: match the `loading` (or `connecting`) status inside the render function and return your own affordance.

**Migration:**

Before:
```tsx
const MoviesView = loader.View(({ data }) => <List items={data.movies} />, {
  fallback: <Spinner />,
});
```

After:
```tsx
const MoviesView = loader.View((s) => {
  switch (s.status) {
    case 'loading':
      return <Spinner />;
    case 'error':
      return <ErrorBox err={s.error} />;
    case 'success':
    case 'revalidating':
      return <List items={s.data.movies} />;
  }
});
```

Reading `reload` (now via `useReload()`):
```tsx
import { useReload } from 'hono-preact';

const StatsView = statsLoader.View((s) => {
  const { reload } = useReload();
  if (s.status === 'loading') return <Skeleton />;
  if (s.status === 'error') return <button onClick={reload}>Retry</button>;
  return <Stats data={s.data} />;
});
```

Before (`loader.Boundary` + `useData()`):
```tsx
<loader.Boundary fallback={<Skeleton />}>
  <Consumer />
</loader.Boundary>;

function Consumer() {
  const { data, loading } = loader.useData();
  if (loading) return <Skeleton />;
  return <Real data={data!} />;
}
```

After:
```tsx
<loader.Boundary>
  <Consumer />
</loader.Boundary>;

function Consumer() {
  const s = loader.useData();
  if (s.status === 'loading') return <Skeleton />;
  return <Real data={s.data} />;
}
```

**Accumulating `.View()` (live loaders):** The render function receives a `StreamState<Acc>` union. The `fallback` option is removed; match `status === 'connecting'` (which carries no data) to render the pre-first-chunk state. `status` encodes `'connecting' | 'open' | 'closed' | 'error'`, which is richer than a boolean and covers SSR, hydration, and mid-session disconnects.

Before:
```tsx
const ActivityBar = loader.View<Acc>(
  ({ data, status }) => <Feed events={data} status={status} />,
  { initial: [], reduce, fallback: <p>Connecting...</p> }
);
```

After:
```tsx
const ActivityBar = loader.View<Acc>(
  (s) =>
    s.status === 'connecting' ? (
      <p>Connecting...</p>
    ) : (
      <Feed events={s.data} status={s.status} />
    ),
  { initial: [], reduce }
);
```

### 2. `defineLoader({ fallbackDelay })` removed

**Affects:** Any `defineLoader` call that passes `fallbackDelay`.

The `fallbackDelay` option (introduced in v0.7 as part of the delayed-fallback feature, PR #136) is removed. It delayed mounting the Suspense fallback on client navigation so a fast response never flashed a spinner on screen.

With state-based loaders there is no longer a Suspense fallback to delay. The same goal is achieved by the stale-while-revalidate behavior: on reload the status is `revalidating`, which still carries the previous `data`, so the page content stays visible and never flashes. On a cold navigation, the `loading` arm renders from the first frame, which is expected.

**Migration:** Remove `fallbackDelay` from all `defineLoader` calls. No other change is needed once you have migrated to the `LoaderState` switch pattern (breaking change 1 above).

Before:
```ts
export const movie = defineLoader(
  async ({ location }) => fetchMovie(location.pathParams.id),
  { fallbackDelay: 200 }
);
```

After:
```ts
export const movie = defineLoader(
  async ({ location }) => fetchMovie(location.pathParams.id)
);
```

### 3. `preact/compat` is no longer loaded by the framework runtime (diff-invisible)

**Affects:** Apps that relied on implicit `preact/compat` behavior injected by the framework's Vite plugin or runtime.

These are behavioral changes that do not appear in the TypeScript API diff. You will not get a compile error, but runtime behavior changes if your app depended on any of the following.

#### 3a. `@preact/preset-vite` `reactAliasesEnabled` is now `false`

The framework's `honoPreact()` Vite plugin previously called `@preact/preset-vite` with the default `reactAliasesEnabled: true`, which aliased `react`, `react-dom`, and `react-is` to `preact/compat` in both source and the bundle. This alias is now **disabled** (`reactAliasesEnabled: false`).

**Impact:** If your app (or a UI library in your app) imports from `react` or `react-dom` and relied on those resolving to `preact/compat` transparently, the import now fails to resolve.

**Migration:** Either add the aliases yourself in your `vite.config.ts`, or replace the `react`/`react-dom` imports with their `preact` equivalents (`preact`, `preact/hooks`).

#### 3b. `preact/compat` removed from Vite `resolve.dedupe`

`'preact/compat'` is no longer in the `resolve.dedupe` list the framework injected. Apps or libraries that relied on this deduplication to ensure a single copy of `preact/compat` is loaded may observe duplicate module instances.

**Migration:** Add `'preact/compat'` to your own `resolve.dedupe` if you need it.

#### 3c. `preact/compat` global renderer `options` patches no longer run

`preact/compat` registers global patches on Preact's `options` object at import time (for `forwardRef`, `PureComponent`, React-event-name mapping, etc.). Because `preact/compat` is never imported by the framework runtime, these patches no longer run unless you import `preact/compat` yourself.

**Impact:** If your components use React-compat APIs (`forwardRef`, `PureComponent`, `useImperativeHandle` via the compat shim) and you did not explicitly import `preact/compat`, they will stop working.

**Migration:** Import `preact/compat` explicitly in your entry point if you need it, or migrate to native Preact equivalents.

### 4. `socketsHandler({ resolvePageUse })` is now required (#178)

`resolvePageUse` on the public `socketsHandler` (`@hono-preact/server`) was optional and silently defaulted to a no-op, which meant a hand-wired socket handler could drop a page's route/layout `use` chain (auth/permission gates) on the WebSocket upgrade path. It is now **required and validated at construction**: `socketsHandler` throws if it is absent or not a function, matching `loadersHandler` / `pageActionsHandler`.

**Impact:** Only affects code that constructs `socketsHandler` directly (a hand-rolled server entry). The framework's generated server entry (`createServerEntry`) already wires `resolvePageUse`, so apps using the standard scaffold are unaffected.

**Migration:** Pass `resolvePageUse: makePageUseResolver(routes).byPath` when constructing `socketsHandler` yourself.

## What's New

### Loader schema validation surfaces field issues (#179)

A loader's `paramsSchema` / `searchSchema` failure now carries the same normalized field issues an action's `deny(422)` does. On client navigation the failure reaches the loader's error boundary as a new `LoaderValidationError` (exported from `hono-preact`); read its issues with `getValidationIssues(error)` — the same reader the action path uses — to render `<FieldError>`-style messages uniformly across actions and loaders. A direct/SSR load of an invalid URL still returns a bare `400`/`404` status. See the [validation docs](https://framework.sbesh.com/docs/validation).
