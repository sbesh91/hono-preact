# hono-preact v0.11.0

This cut is a **developer-experience and first-load-performance wave**. Most of
it comes out of the #260 ergonomics review (public-surface features, dev-seam
fixes, template refresh), landing alongside two first-load performance features
(route modulepreload + CSS auto-split), a six-finding security pass, a
realtime auth-hardening fix (`serverRoute(r).socket` / `.room` now really bind
their route), and one breaking UI change.

## Versions

| Package | From | To | Why |
|---|---|---|---|
| `hono-preact` | 0.10.1 | **0.11.0** | features + breaking realtime/auth fixes (lockstep) |
| `create-hono-preact` | 0.10.1 | **0.11.0** | template/CLI refresh (lockstep) |
| `hono-preact-ui` | 0.2.0 | **0.3.0** | breaking `Select`/`Combobox` props |

`hono-preact` + `create-hono-preact` move in lockstep on the minor;
`hono-preact-ui` cuts independently and takes a breaking bump of its own.

## Breaking changes

### 1. `hono-preact-ui`: discriminated `Select` / `Combobox` selection props (#268)

`SelectionProps<Value>` is now a **union discriminated on `multiple`**, and the
empty-controlled state is spelled `null` instead of an ambiguous `undefined`:

- **Single (default):** `value?: Value | null`,
  `onValueChange?: (v: Value | null) => void`.
- **Multiple (`multiple: true`):** `value?: readonly Value[]`,
  `onValueChange?: (v: Value[]) => void`.

`null` means **controlled-and-empty**; `undefined` stays **uncontrolled**. Clear
is now observable: `Combobox.Clear` and form reset emit `null` (single) / `[]`
(multi) instead of silently swallowing the change (the internal lying casts at
those sites are gone).

**Migration:**

- A single-select consumer that modeled "cleared" as `value={undefined}` while
  otherwise controlling `value` should switch to `value={null}`; leave `value`
  off entirely only if the component is meant to be uncontrolled.
- A `onValueChange` handler for a single select now receives `Value | null`
  (handle the clear case).
- Custom roots built on `useListboxSelection` must move to the reshaped
  `values` / `setValues` API (`readonly Value[]`). The exported
  `normalizeSelectionProps` helper maps either props arm onto the internal
  `readonly Value[]` state.

### 2. `hono-preact`: SSE error frames are masked in production (#263)

`event: error` SSE frames previously shipped the raw `err.message` / `err.name`
with no gate (the inverse of the JSON path's policy). Production now sends a
generic `Stream failed` (name `Error`); **dev** passes the real message through.

This changes the **production wire payload** for mid-stream errors. No client
change is needed, the decoder handles any message, but anything asserting on the
exact prod error text over the wire will see `Stream failed`.

### 3. `hono-preact`: bare actions get no page tier (#261, security finding 2)

A **bare action** (colocated, not route-bound) now runs with an empty page-use
chain (`EMPTY_PAGE_USE`), identical to a bare loader, instead of a page-use chain
fuzzy-matched from the client-sent URL. This closes a guard-divergence hole (the
client could steer the match to a weaker sibling route). If you relied on a bare
action inheriting a route's page tier, **bind it to the route**
(`serverRoute(r).action(...)`) or put the guard on the unit via a per-action
`use`. See Security below.

### 4. `hono-preact` default flip: CSS minifier is Lightning CSS (#266)

With the new CSS pipeline, `cssMinify` now defaults to Lightning CSS (the
umbrella `lightningcss` dependency comes along). Output is functionally
equivalent; if you had explicitly opted into Lightning CSS you can drop that, and
if you need the old esbuild minifier you can still set it explicitly.

### 5. Node engines requirement

`hono-preact` and `create-hono-preact` now declare
`engines.node: "^22.18.0 || >=24.11.0"`. Older Node lines are no longer
supported.

### 6. `hono-preact`: `serverRoute(r).socket` / `.room` now really bind their route (fail-closed) (#274, closes #273 item 1)

Previously the route argument was silently discarded: upgrade-guard resolution
came only from the module mount, so a route-bound socket/room in a `src/server`
registry module ran with NO page-tier gates, and a declared route that
disagreed with the module mount was silently ignored. Now the declared pattern
is stamped on the def, validated at boot exactly like route-bound
loaders/actions (mismatched, childless-wildcard, and unknown-pattern bindings
fail the boot), takes precedence in upgrade-guard resolution, and the
`/__sockets` endpoint refuses to upgrade (500) until the binding check passes.
Registry-module sockets/rooms bound to a route now run that route's gates:
connections that previously succeeded may now be denied, which is the fix
working. One direction runs FEWER gates by design: a subtree spelling on a
route-attached module (`serverRoute('/x/*').socket`/`.room`) now selects the
layout-scope (subtree) chain instead of the exact page chain the
discarded-route behavior fell back to, so a guard living only on the index
child's own `use` no longer covers that socket/room; bind the exact path to
keep the index page's full gate chain. Bare `defineSocket`/`defineRoom` are
unchanged. Not visible in an export-surface diff (no public export changes),
so this manual entry is load-bearing.

### 7. `hono-preact`: a route-BOUND room's channel congruence now fails the boot; a COLOCATED room's does not

A room's upgrade guard reads `ctx.location.pathParams` from the room's CHANNEL
key, not from a URL. So a room sitting on a route whose params the channel does
not carry ran its guard with those params `undefined`: a guard reading
`pathParams.id` either threw (500 on every join) or authorized against
`undefined`.

Only an explicitly bound `serverRoute(r).room` (shipped in v0.10.1) can fail
its boot on this mismatch, and only when at least one of the three guard tiers
that read the same `pathParams` is non-empty: the app tier
(`defineApp({ use })`), the page tier (route or layout `use`), or the room's
own `use`. So an app whose BOUND room sits on a param-bearing route with a
non-matching channel (for example `/board/:id` next to
`defineChannel('board/:boardId')`), guarded on any of those tiers, booted on
v0.10 and now fails to boot. It fails loud and closed, naming the room, the
route, the missing param, and the channel.

A COLOCATED room (a `.server.ts` sibling, no explicit `serverRoute(...).room`
binding, shipped in v0.9) NEVER fails the boot on this mismatch, regardless of
guard-tier liveness: colocation predates the route-bound param model, and the
room's "route" is only its module's file mount, not an authored binding, so
throwing here would break every released colocated-room app the moment it grew
a guard anywhere in that chain. A bound room with all three guard tiers empty
today is exempted the same way. Both cases instead fire a loud console
advisory, in BOTH development AND production (not dev-only), naming the room,
the route, the missing param, and the channel: a guard on the route, live now
or added later, reads the missing param as `undefined` either way, so the
warning does not go away just because nothing reads it today.

Two fixes for a throwing (bound) room: rename the channel or route param so
the names match, or move the room into a `src/server` registry module with no
`serverRoute` binding (a bare registry room is route-independent and is not
congruence-checked). A colocated room never needs a fix to keep booting (it
never throws); the advisory is the signal to fix the mismatch, or bind
explicitly with `serverRoute(...).room(...)` if the room genuinely needs to
authorize on that route param. A room on a param-less route is unaffected
either way.

### 8. `hono-preact`: a param-bearing `serverRoute(r).socket` now requires `params`

`serverRoute(r).socket` shipped in v0.10.1 but **discarded the route argument**, so
a socket bound to `/board/:id` ran with no route gates at all. That is the auth hole
this release closes, and closing it breaks the released spelling.

A bound socket on a param-bearing route now carries a typed `params` option, so
`useSocket(ref)` and `ref.useSocket({ onMessage })` no longer compile: pass the
route params, `useSocket(ref, { params: { id } })`. A client that forces past the
types is denied (close `4403`) on every connection for the missing slot. The params
are validated at the upgrade, read by the route's guard as `ctx.location.pathParams`,
and passed to the edge `data` factory as its second argument.

A **colocated** socket (a `.server.ts` sibling with no `serverRoute` binding) has no
param wire, keeps empty `pathParams`, and is unaffected.

### 9. `hono-preact`: a route-bound socket or room on a non-conforming `:param` route fails the boot

The framework's param-name grammar is `[A-Za-z0-9_]+`, but the underlying route
matcher is wider and will bind, for example, `/board/:board-id`. For a route-BOUND
socket or room, that mismatch is unsafe: the framework sees no param there, so the
socket requires nothing, resolves empty params, never denies, and hands its guard an
empty param the type contract promised it would receive. That silent degradation is
now a boot error naming the unit, the route, and the offending segment.

Ordinary HTTP routes, loaders, actions, and colocated (unbound) sockets and rooms on
the same route are deliberately unaffected, so an app that merely serves such a route
keeps booting. Fix: rename the route param to the supported class, or drop the
`serverRoute(...)` binding on that unit.

### 10. `hono-preact`: `defineChannel` rejects a non-conforming param name (narrower than "any colon")

`defineChannel` (shipped v0.8.0) now throws at definition time in two cases:

- A `:`-prefixed segment whose name carries a character outside
  `[A-Za-z0-9_]` (`:board-id`, a hyphen).
- A colon-namespaced LITERAL segment (the colon is not at the segment's
  start) whose suffix parses as a bare `[A-Za-z0-9_]+` identifier:
  `board:boardId`, `board-:id`, `metrics:cpu`.

Both shapes were already broken, silently and dangerously: the type layer read
them as a required param, while the runtime interpolator did not recognize
one, so the key was dropped and the topic collapsed to a **constant string**.
Every resource then shared one topic, so presence rosters and broadcasts
cross-leaked between them. The throw surfaces a pre-existing bug rather than
adding a restriction.

A colon-namespaced literal whose suffix is NOT a bare identifier does **not**
throw and keeps working exactly as before: `notifications:user-alerts`,
`chat:lobby-1`, `events:order.created`. The type layer never claimed a param
there either (a hyphen or a dot fails the same character class), so these
remain ordinary, working constant names, not a hazard.

Fix (for a rejected spelling): rename the param to the supported class
(`board/:boardId`).

### 11. `hono-preact`: `useRoom` enforces its `key` contract at the type level

`useRoom`'s room-key option was typed loosely enough to be sidestepped two ways,
both of which now fail to compile:

- **A stray `key` on a param-less channel.** `KeyOption` spelled the no-params
  branch `{ key?: {} }`, and TypeScript's `{}` structurally accepts any non-nullish
  value, so `useRoom(ref, { key: { anything: 'x' } })` type-checked against a
  channel with no `:params`. The branch is now `{ key?: never }`.
- **Omitting the options object entirely.** `opts` was an optional parameter, so
  `useRoom(ref)` compiled even for a channel whose `key` is required. The options
  argument is now a conditional rest tuple, required exactly when the channel has
  params. The `serverRooms.x.useRoom(...)` ref-method form is fixed the same way,
  from one shared definition.

Both spellings only ever produced broken code: a key handed to a param-less channel
is interpolated into nothing, and a param-bearing room opened with no key resolves
no topic and is denied (`4403`) at the upgrade. This turns two guaranteed runtime
failures into compile errors. `useSocket`'s `params` option carries the identical
contract.

### 12. `hono-preact`: a `+` (rest-one-or-more) `:param` slot is now required

v0.10.1's room-key required-slot check explicitly excluded `+`
(`flag !== '?' && flag !== '*' && flag !== '+'`), so a channel like
`defineChannel('files/:rest+')` connected fine with no `rest` key at all:
`interpolatePattern` drops the missing segment, and the connection silently
joined the truncated topic `files` instead of denying. The shared
`requiredParamSlots` helper this release introduces excludes only `?` and
`*`, correctly treating `+` as required, matching the type-level contract
(`+` always typed as required) and the runtime route matcher (which refuses
to match a `+` segment with no value).

A released room bound to a `+` channel, connected by a hand-rolled client that
omits the rest key, previously joined the truncated topic silently and now
gets a `4403` deny instead. Sockets are unaffected by this specific
regression (route-bound socket params are new in this release and always
treated `+` as required), but the rule is now identical for both. Fix: always
supply the rest-one-or-more param, or relax the channel/route to `*`
(rest-zero-or-more) if an empty value is legitimate.

### 13. `hono-preact`: `useSocket`/`useRoom` moved from `(ref, opts?)` to a conditional rest tuple

Both hooks shipped in v0.8.0 with a plain optional second parameter. This
release switches both to a rest tuple (`UseSocketArgs<R>`/`UseRoomArgs<R>`),
so the options argument is itself required exactly when the bound
route/channel has params. Previously `useSocket(boundRef)` with the options
argument omitted entirely compiled even though a param-bearing binding
required `params` once an options object was actually passed; the type hole
only bit at the point a real options object was supplied. (Entry 11 above
covers the identical change for `useRoom`'s `key`.)

A generic wrapper written against the old signature no longer type-checks,
independent of whether the wrapped ref happens to have params:

```ts
function f<R extends SocketRef<unknown, unknown>>(
  ref: R,
  opts?: UseSocketOptions<R>
) {
  return useSocket(ref, opts); // TS2345: not assignable to the rest tuple
}
```

`UseSocketArgs`/`UseRoomArgs` are now exported from the public barrel
(`hono-preact`) precisely so a wrapper like this can forward the tuple by
name instead of re-deriving it:

```ts
function f<R extends SocketRef<unknown, unknown>>(
  ref: R,
  ...args: UseSocketArgs<R>
) {
  return useSocket(ref, ...args);
}
```

The identical shape applies to `useRoom`/`UseRoomArgs`.

### 14. `hono-preact`: `defineRoom` now re-validates a hand-rolled `Channel`

`Channel` is a public type export (shipped with `defineChannel` in v0.8.0), so
a released app could construct one directly as a `{ name, key }` object
literal without ever calling `defineChannel`, bypassing its name-conformance
check entirely. `defineRoom` and `serverRoute(r).room` (both v0.8.0) now run
the SAME check on the `channel` argument, so a hand-rolled `Channel` whose
`name` carries a non-conforming `:`-segment throws at the `defineRoom(...)` /
`serverRoute(r).room(...)` call instead of silently collapsing every
connection onto one degenerate constant topic (cross-resource presence and
broadcast leak). A released app that constructed such a `Channel` by hand,
and would previously boot with every connection sharing one topic, now fails
to define the room. Fix: construct the channel with `defineChannel` (which
already validated the name), or rename the offending segment to the
supported `[A-Za-z0-9_]` class.

### 15. `hono-preact`: realtime params are restricted to their pattern's declared slots

A room's `onJoin` `ctx.params` and a room/socket guard's `pathParams` are now
restricted to the channel/route pattern's own declared `:param` slots; a key
on the `r=` wire that the pattern does not declare is silently dropped
rather than passed through. Both `resolveRoomKey` (rooms, shipped v0.8.0)
and `resolveSocketParams` (bound sockets, new this release) share this
restriction via `parseKeyParams`. A released room that (knowingly or not)
relied on an extra wire key reaching `onJoin`'s `params` or a guard's
`pathParams` no longer sees it there. This closes a client-injection surface
(a connection could otherwise smuggle an arbitrary extra key onto the wire
that no real page navigation could ever produce), so the pre-break behavior
was unsafe. Fix: stop relying on undeclared keys; pass real data through the
`data` factory (edge-derived, not wire-derived) instead.

### 16. `hono-preact`: a room `r=` key wire that is present but not a JSON object now denies

`resolveRoomKey` shipped in v0.8.0 and, from that same version, coerced any
non-plain-object parse result of the `r=` query value (`null`, a number, a
string, an array) to `{}` instead of denying. That meant a garbage `r=` value
on a PARAM-LESS channel (nothing required, so the coerced-to-`{}` params
still satisfied the "no required slots" check) connected successfully; only
a param-bearing channel denied, and only via the ordinary missing-params
path. The shared `parseKeyParams` this release introduces treats a
present-but-non-object payload as a contract lie (not a missing param) and
denies it outright, `4403`, regardless of whether the channel has any
required slots at all.

A released room connected by a hand-rolled or buggy client that sent, say,
`r=null` or `r="x"` on a param-less channel previously connected; it is now
denied. Fix: send a well-formed JSON object for `r=`, or omit it entirely
for a param-less channel.

### 17. `hono-preact`: `defineChannel`/`defineRoom` reject a channel name with more than one optional/rest slot

`interpolatePattern` drops an absent-or-empty segment (correct for a route
href, where a trailing optional legitimately shortens the URL, and
unchanged by this entry). For a CHANNEL name that is a hazard once TWO OR
MORE optional (`?`) or rest (`*`/`+`) slots exist: two distinct key sets
that each omit a DIFFERENT slot collapse onto the SAME topic. For example
`defineChannel('room/:a?/:b?')`: key `{ a: 'x' }` and key `{ b: 'x' }` both
resolve to the topic `room/x`, cross-leaking presence and broadcasts between
what the author modeled as two distinct resources.

`defineChannel` (v0.8.0) now throws at definition time for a name carrying
more than one such slot, and `defineRoom`'s hand-rolled-`Channel`
re-validation path (entry 14 above) rejects it the same way. A released
channel spelled with two or more optional/rest slots, previously silent and
dangerously ambiguous, now fails to define instead of booting into a shared
constant topic. A channel with AT MOST ONE optional/rest slot is unaffected.
Fix: use at most one optional or rest slot in a channel name, or make the
extra slot(s) required.

### 18. `hono-preact`: every guard-facing params map is now a null-prototype object

A room's `onJoin` `params` (shipped v0.8.0 through v0.10.1, built with
`Object.fromEntries`, which inherits `Object.prototype`) and the loader/guard
`ctx.location.pathParams` / `ctx.location.searchParams` on the loader RPC
path are now built with the shared `toNullProtoParams` helper, so they carry
no prototype at all, not even `Object.prototype`.

The param-name grammar admits every `Object.prototype` member name
(`constructor`, `toString`, `valueOf`, `hasOwnProperty`, `isPrototypeOf`,
`propertyIsEnumerable`, `toLocaleString`). A `params` object that inherits
`Object.prototype` lets a guard reading a MISSING param named after one of
those members resolve the inherited (truthy) member instead of `undefined`:
`if (!params.constructor) deny()` on a route or channel bound to
`/plugin/:constructor` would wrongly pass, because `params.constructor`
resolved the inherited `Object` constructor function rather than reading as
absent. Null-prototype closes that off at the source, for every param name,
not just the ones a denylist would think to name.

A released handler that called an inherited `Object` method directly on one
of these params objects -- `params.hasOwnProperty(k)`, `params.toString()`,
`params.valueOf()`, or similar -- now throws `TypeError:
params.hasOwnProperty is not a function` (there is no inherited method left
to call). Fix: use `Object.hasOwn(params, k)` for a presence check, `k in
params`, or destructure the params object directly (`const { id } =
params`); none of these ever depended on the prototype chain, so they behave
identically before and after this change.

## New features

### Realtime

- **Route-bound sockets authorize on route params (#273 item 1 follow-on).**
  `serverRoute('/board/:id').socket(handler)` now gives the socket a typed param
  wire: `useSocket(ref, { params })` is required and typed from the route pattern,
  the values are validated at the upgrade (a missing param denies the connection,
  close `4403`), and they reach both the route's page-`use` guard as
  `ctx.location.pathParams` and the edge `data` factory as its second argument
  (`data: (c, params) => ...`). A **colocated** socket (no `serverRoute` binding)
  has no param wire and keeps empty `pathParams`, so explicit binding is the way
  to authorize a socket on a route param. Realtime params are also now restricted
  to their pattern's declared slots, so a client cannot inject an undeclared key
  into a guard's `pathParams`.

### First-load performance

- **Route-level modulepreload (#250, #252).** The SSR document now preloads the
  client entry's static-import closure (#250) and the matched route's own chunk
  (#252) as `<link rel="modulepreload">` hints (route own-chunk at
  `fetchpriority=low`), plus closure-only `Link` response headers. This removes
  the 3-4 hop discovery chain to a route's code, hydrating the route ~1s earlier
  on mid-tier mobile at real latency. Universal across the Cloudflare and Node
  adapters; no bundle patching. (This is the productionized #121, now closed. CF
  103 Early Hints promotion is tracked separately in #270; it does **not** apply
  to `modulepreload` headers, so this cut claims the modulepreload win but not an
  Early-Hints win.)
- **CSS auto-split (#266).** Author **one** global stylesheet and the framework
  tree-shakes it at build time into a residual global sheet plus per-chunk route
  sheets, delivered through the existing route-CSS machinery:

  ```ts
  honoPreact({
    adapter: cloudflareAdapter(),
    css: { global: 'src/styles/root.css' }, // autoSplit defaults true, minSize 1024
  })
  ```

  SSR head order is font preloads → modulepreloads → user head → global sheet →
  route sheets. Dev links the source stylesheet byte-for-byte with the old
  hand-authored `?url` link, so dev behavior is unchanged. A rule rides a route
  chunk only when it is provably chunk-scoped; anything unprovable stays in the
  global sheet.

### Public surface & DX (from #260)

- **Head hooks re-exported (#264).** `useTitle`, `useTitleTemplate`, `useMeta`,
  `useLink`, `useLang`, `useScript` (and their option types) are now exported
  from `hono-preact`. No more importing `hoofd` by a name you never installed.
  New Head Management docs page. (`hoofd` remains an external peer, now declared
  at `>=1.3.0`.)
- **`createCaller` covers streaming (#264).** Overloads extend to streaming
  loaders and actions and gain `signal` support for aborting streams in tests.
  `createCaller` is now the advertised test entrypoint (with `isDeny` /
  `isRedirect`), replacing `as unknown as` casts and `__outcome` probing.
- **Typed channel `eventStream` (#264).** Typed server-to-client channel
  event streaming reachable without workarounds.
- **Public `bootClient()` + stated `clientEntry` contract (#265).** The history
  shim, nav-transition scheduler, and stream-registry installs are now a public
  `bootClient()` export (from `@hono-preact/iso`, re-exported at the root). A
  custom `clientEntry` that never calls it warns in dev.
- **Config-probe watchers (#265).** Creating or deleting `src/api.ts`,
  `src/app-config.ts`, or the `src/server/` registry now restarts the dev server
  automatically instead of silently degrading until a manual restart.

### Routing

- **Wildcard subtree binder (#267).** `serverRoute('/demo/projects/*')` binds a
  **layout scope**: the node's composed chain (all ancestor gates plus its own)
  without any child's additions, giving a spelling for "the layout's chain,
  without the index child's guards" that the shared layout/index pattern string
  could not express. Ships with an aliasing diagnostic.
- **Hash-aware transition classification (#269, closes #219).** The nav
  scheduler now classifies navigations by pathname + search; **hash-only URL
  changes are not navigations** (no view transition, no skip needed), fixing the
  TOC/permalink full-page-transition flash that app code could not reach.
  `skipNextNavTransition(target?)` is now **URL-keyed** and self-expiring, so the
  `NavLink` skip-strand (#219) is structurally impossible; the no-arg form keeps
  legacy wildcard semantics.
- **Root `/` join unified (#272).** The root path join is now consistent across
  the runtime walkers and typed routes.

### Scaffolding (`create-hono-preact`, #262)

- `typecheck` is now a first-class template script (`tsc --noEmit`); the scaffold
  integration test typechecks generated projects for both adapters.
- Template routes teach current idioms: the obsolete explicit `server:` field is
  gone (auto-discovery since 0.10.0), typed routing set up the way `apps/site`
  does it.
- `{{name}}` is substituted into source files, so first `pnpm dev` no longer
  renders a literal `{{name}}` title.
- Windows fix: the package manager and git are spawned through a shell (#257).

## Security (#261)

Six findings from the 2026-07-08 review of `packages/` (`iso`, `server`, `vite`,
`create-hono-preact`), each with a regression test:

| # | Severity | Fix |
|---|---|---|
| 1 | HIGH | **create-hono-preact RCE.** Strict project-name slug validation (`^[A-Za-z0-9][A-Za-z0-9._-]*$`) before any filesystem/scaffold work, on both the flag/stdin and prompt paths. Rejects quotes (JSON breakout into a sibling `postinstall`) and shell metacharacters. Defense-in-depth: `package.json` `name` set structurally so `JSON.stringify` escapes it. |
| 2 | HIGH | **Bare-action guard divergence.** Bare actions now get **no page tier** (`EMPTY_PAGE_USE`), identical to bare loaders, instead of a fuzzy URL-matched page-use chain the client could steer to a weaker sibling route. See Breaking #3. |
| 3 | HIGH | **Realtime identity spoofing.** Strips the whole server-controlled `x-hp-*` header namespace from the forwarded upgrade before stamping, on both the room and socket-forward branches, so a client `x-hp-data` can't forge `conn.data` / `socket.data` when the data factory returns `undefined`. |
| 4 | HIGH | **Server-code leak via dynamic import.** Stubs no-substitution template-literal `.server` specifiers (`` import(`./x.server.js`) ``) and **fails the build closed** on an interpolated `.server` dynamic import that can't be stubbed. |
| 5 | HIGH | **Guard-strip namespace bypass.** Matches namespace-import member calls (`import * as hp; hp.defineServerMiddleware(...)`) as well as named imports, so route-level middleware bodies never ship to the client. |
| 6 | MEDIUM | **create-hono-preact `deploy`-script injection.** Same slug validation as #1 closes the quote-free variant. |

**Design note on finding 2:** binding a bare action to its colocation route's
guard chain was **rejected** as contrary to route-independence (a bare
`defineAction` has no route to bind to). The fix aligns actions with the loaders
reference implementation, bare units are route-independent and guarded at the
unit level (`defineAction({ use })`) or via `serverRoute(r).action`. The unsafe
`resolvePageUseByPath` path (and the underlying `makePageUseResolver.byPath`
fuzzy matcher) is removed rather than left as an attractive nuisance.

## Upgrade checklist

1. **Node:** ensure `^22.18.0 || >=24.11.0`.
2. **hono-preact-ui consumers:** migrate `Select`/`Combobox` cleared state to
   `null`, handle `Value | null` in single-select `onValueChange`, and move
   custom roots to `useListboxSelection`'s `values`/`setValues`.
3. **Bare actions** relying on a route page tier: bind them with
   `serverRoute(r).action(...)` or add a per-action `use`.
4. **Route-bound sockets/rooms:** a registry-module `serverRoute(r).socket` /
   `.room` now enforces that route's gates and fails the boot on a misbinding
   (mismatched, childless-wildcard, or unknown pattern). Confirm each binding
   resolves to a real route, and use the **exact** path (not `/*`) when you need
   the index page's full gate chain rather than the layout-subtree chain.
5. **Prod SSE error assertions:** expect `Stream failed`.
6. **CSS:** optionally drop explicit Lightning CSS opt-in (now the default); adopt
   `css: { global }` auto-split if you want route-scoped delivery from one sheet.
7. **Head hooks:** import from `hono-preact` instead of `hoofd` directly.
