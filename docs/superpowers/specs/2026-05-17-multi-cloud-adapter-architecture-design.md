# Multi-Cloud Adapter Architecture — Design

Date: 2026-05-17
Status: Approved (pending spec review)

## Summary

Replace the hardcoded Cloudflare build/dev toolchain in `honoPreact()` with a
deployment **adapter** abstraction. An adapter is an object passed to
`honoPreact({ adapter })` that owns everything platform-specific: which Vite
plugins drive build and dev, and how the framework-generated Hono app is
adapted to the target runtime. This spec delivers the adapter interface plus
two concrete adapters — Cloudflare and Node — and verifies WebSocket support
in dev on both.

The reference for the pattern is [pracht](https://github.com/JoviDeCroock/pracht)
(`docs/ADAPTERS.md`): a platform-agnostic core, thin per-platform adapter
packages, each generating the server entry for its target.

## Motivation

`hono-preact.ts` hardcodes `@hono/vite-build/cloudflare-workers` and the
`@hono/vite-dev-server` Cloudflare adapter (lines 1-3, 132-151). The framework
is therefore Cloudflare-only in practice, even though Hono apps run on every
JavaScript runtime. Two concrete problems motivate the change:

1. **No multi-cloud deploys.** Users cannot target Node, Bun, or Deno even
   though the generated server entry (a plain Hono app) is runtime-agnostic.
2. **WebSocket upgrades do not work in `vite dev`.** `@hono/vite-dev-server`
   runs requests through a Connect middleware and never sees the HTTP server's
   `upgrade` event. This was documented as a known gap in
   `docs/superpowers/research/2026-05-14-hono-primitives-audit.md` and deferred
   past v0.1.

Both are resolved by making the deployment target an explicit, swappable
adapter, and by building each adapter on Vite's Environment API so its dev
runtime is a true production mirror.

## Goals

- Define a stable `HonoPreactAdapter` interface with a minimal surface.
- Ship a Cloudflare adapter (built on `@cloudflare/vite-plugin`, workerd dev).
- Ship a Node adapter (built on Vite's native Environment API, in-process dev).
- WebSocket-in-dev works and is verified on both adapters.
- `apps/site` migrates onto the Cloudflare adapter with no behavior change.
- A minimal `apps/example-node` exercises and documents the Node adapter.

## Non-Goals

- Bun, Deno, Vercel, or other adapters. The interface must not preclude them,
  but they are out of scope for this spec.
- SSG/ISG or any render-mode work.
- A framework-provided WebSocket abstraction (see WebSockets section).
- Changes to loaders, actions, routing, or guards.

## Architecture

### Entry-generation split

Server-entry generation today is one monolithic `generateServerEntrySource()`
in `packages/vite/src/server-entry.ts`. It splits into two layers:

- **Core app module (platform-agnostic, framework-owned).** Generates the Hono
  app: `loadersHandler` / `actionsHandler` / `renderPage` and the optional
  `.route('/', userApp)` mount. Its job ends at `export default app`. Written
  to disk under `node_modules/.vite/hono-preact/` as today.
- **Entry wrapper (platform-specific, adapter-owned).** Generated by the
  adapter's `wrapEntry()`. Imports the core app module and adapts it to the
  target runtime. Written to disk as a second file; this wrapper is the entry
  the adapter's build/dev plugins consume.

This mirrors pracht's `virtual:pracht/server` (adapter-generated, importing the
core resolved app).

### The adapter interface

Lives in a framework-owned type module, re-exported for adapter authors.

```ts
interface HonoPreactAdapterContext {
  root: string;              // Vite project root
  command: 'serve' | 'build';
  coreAppModuleId: string;   // absolute path of the core-generated Hono app module
  outDir: string;
}

interface HonoPreactAdapter {
  name: string;                                          // 'cloudflare' | 'node'
  vitePlugins(ctx: HonoPreactAdapterContext): Plugin[];  // terminal build + dev plugins
  wrapEntry(ctx: HonoPreactAdapterContext): string;      // platform tail source
}
```

The interface is deliberately two methods. Anything an adapter needs to
influence — environment/build config, dev middleware, the `upgrade`-event
hook — is carried by the plugins `vitePlugins()` returns, since Vite plugins
already expose `config`, `configureServer`, and build hooks. No capability
needs a dedicated interface slot.

### How `honoPreact()` consumes it

`adapter` becomes a **required** option of `HonoPreactOptions`. Everything else
(`layout`, `routes`, `api` path defaults) stays zero-config. The plugin chain
becomes:

```
[ framework plugins: config, client-shim, client-entry, server-entry,
  server-loader-validation, module-key, server-only, guard-strip ]
+ ...adapter.vitePlugins(ctx)
+ ...preact()
```

The hardcoded `@hono/vite-build` and `@hono/vite-dev-server` instances
(`hono-preact.ts:132-151`) are deleted. The `server-entry` plugin writes the
core app module; it also writes `adapter.wrapEntry(ctx)` as the entry the
adapter's plugins build/serve.

### `configPlugin` gutting

The current `configPlugin` (`hono-preact.ts:50-112`) encodes the legacy
`mode === 'client'` two-pass build model. Both adapters use Vite's Environment
API instead, so this branching is removed. The framework `configPlugin` shrinks
to only genuinely shared config (`preact` dedupe, `target: 'esnext'`,
`assetsDir`). Each adapter contributes its own `environments` / `builder`
config through a `config` hook on one of its `vitePlugins()`.

Framework plugins that branch on client-vs-server must be audited for the
discriminator change: `serverOnlyPlugin` uses `options.ssr` (environment-safe,
no change); `client-shim`, `client-entry`, `module-key`, and `guard-strip`
each need a check that they key off environment/`ssr` rather than `mode`.

## The Cloudflare adapter

`hono-preact/adapter-cloudflare`, factory `cloudflareAdapter()`.

- **`vitePlugins()`** returns `[cloudflare()]` from `@cloudflare/vite-plugin`.
  That single plugin drives both dev (workerd) and build, via the Environment
  API. It replaces both `@hono/vite-build/cloudflare-workers` and
  `@hono/vite-dev-server`.
- **`wrapEntry()`** is near-identity: a Hono app's default export is already a
  valid Workers `{ fetch }` handler, so the tail is
  `export { default } from '<coreAppModuleId>';`.
- **`wrangler.jsonc`** stays user-owned; `@cloudflare/vite-plugin` reads it and
  generates the deployment config at build time.
- **WebSocket-in-dev** works for free: dev requests execute in workerd, so
  `upgradeWebSocket` from `hono/cloudflare-workers` and `WebSocketPair` behave
  exactly as in production.
- **Build** is a single `vite build`. The current two-pass
  `vite build --mode client && vite build` is retired.
- Optional peer deps: `@cloudflare/vite-plugin`, `wrangler`.

## The Node adapter

`hono-preact/adapter-node`, factory `nodeAdapter()`. Built entirely on Vite's
native Environment API — it depends on **no** `@hono/vite-*` build/dev tooling.

- **`vitePlugins()`** returns framework-owned plugins:
  - A `config`-hook plugin contributing `environments` (`client` + `ssr`) and a
    `builder` so one `vite build` produces both the browser bundle and the Node
    server bundle.
  - A dev plugin whose `configureServer` hook connects a module runner to the
    `ssr` environment, loads the entry wrapper, pipes requests through
    `app.fetch`, and attaches an `upgrade` listener to `server.httpServer` (via
    `@hono/node-ws`'s `injectWebSocket`) for WebSocket-in-dev. Vite's module
    runner provides server-module HMR.
- **`wrapEntry()`** composes an *outer* Hono app, because Node has no CDN asset
  layer: `serveStatic` (from `@hono/node-server/serve-static`) for `dist/static`,
  then `.route('/', coreApp)` for everything else, booted with `serve()` from
  `@hono/node-server`. The bundled result, `dist/server/server.js`, is the
  runnable production entry (`node dist/server/server.js`).
- **WebSocket-in-dev** is delivered by the `upgrade`-event wiring in the dev
  plugin above.
- Optional peer deps: `@hono/node-server`, `@hono/node-ws`. (These are runtime
  dependencies of the generated entry and the dev plugin, not build tooling.)

## Entry-generation refactor

In `packages/vite/src/server-entry.ts`:

- `generateServerEntrySource()` becomes `generateCoreAppModule()`, emitting only
  the platform-agnostic Hono app ending at `export default app`. Its body
  changes little — it already produces that shape.
- The `server-entry` plugin writes two files under
  `node_modules/.vite/hono-preact/`: the core app module and the adapter's
  `wrapEntry()` output.
- `findApiCatchAllRoutes()` and the catch-all warnings are platform-agnostic
  and unchanged.
- **The `entry` escape-hatch option is removed.** With the adapter owning the
  platform tail, a user-supplied monolithic entry has no coherent plug-in
  point. The `useGeneratedEntry` branching (`hono-preact.ts:47-48, 118-127`) is
  deleted; generation is unconditional. Full control is available by writing a
  custom adapter, now a first-class extension point. This is a small breaking
  change to the public `HonoPreactOptions` API, acceptable post-v0.1.

The `reference_hono_vite_build_entry` constraint (the generated entry had to be
written to disk because `@hono/vite-build` resolves entries via
`import.meta.glob`, which cannot match `virtual:*` ids) no longer applies once
`@hono/vite-build` is gone. The entry files may stay on disk for simplicity;
moving them to virtual modules is a possible later cleanup, out of scope here.

## Packaging

Per the decision to keep the consolidated umbrella, adapters ship as **subpath
exports** of the `hono-preact` umbrella, not separate packages:

- New subpaths `hono-preact/adapter-cloudflare` and `hono-preact/adapter-node`,
  fed through the existing `packages/hono-preact/scripts/consolidate.mjs`.
- Adapter toolchains are declared as **optional `peerDependencies`** of the
  umbrella (`peerDependenciesMeta.*.optional = true`): `@cloudflare/vite-plugin`
  and `wrangler` for the Cloudflare adapter; `@hono/node-server` and
  `@hono/node-ws` for the Node adapter. They must not be hard dependencies, or
  every install drags in both adapters' toolchains.
- Each adapter module imports its toolchain inside a `try`/`catch` and throws a
  clear "install X to use the <name> adapter" message if a peer is missing.
- `packages/vite/package.json`: `@hono/vite-build` and `@hono/vite-dev-server`
  are removed from `peerDependencies` entirely; the four optional peers above
  are added.
- Docs state, per adapter, exactly which peer deps to install.

## WebSockets

WebSocket handler code is **inherently runtime-specific** in Hono:
`upgradeWebSocket` from `hono/cloudflare-workers` versus `@hono/node-ws`'s
`createNodeWebSocket().upgradeWebSocket`. The framework does **not** provide a
unifying `hono-preact/websocket` re-export — that would contradict the
"use Hono as normal; the framework should not pre-decide for users" principle
from the hono-primitives audit, and the runtime handler shapes are not
identical enough to wrap cleanly.

Therefore each target uses its adapter's native WS primitive directly, and the
docs state plainly that WS handler code is per-runtime.

Verification:

- **Node WS:** a `/ws` route in `apps/example-node`, plus an integration test
  that connects a WebSocket client to the `vite dev` server and asserts the
  `upgrade` listener wired by the Node adapter's dev plugin works.
- **Cloudflare WS:** an integration test in `packages/vite/src/__tests__/` that
  boots the `@cloudflare/vite-plugin` (workerd) dev server and connects a
  WebSocket client. No WS route is added to `apps/site`; the production site
  stays clean.

## The Node example app

A new minimal `apps/example-node`, configured with `nodeAdapter()`:

- A couple of SSR routes, one loader, one action, one WebSocket route.
- Builds with `vite build`, runs with `node dist/server/server.js`, dev with
  `vite dev`.
- Serves as both the Node-adapter test surface and the Node-target docs
  reference.

`apps/site` remains the production Cloudflare app and gains no demo routes.

## `apps/site` migration

- `vite.config.ts`: `honoPreact()` → `honoPreact({ adapter: cloudflareAdapter() })`.
- `package.json`: `build` collapses to `vite build`; `preview` becomes
  `vite preview` (workerd-backed); `@hono/vite-build`, `@hono/vite-dev-server`,
  and `miniflare` are dropped from devDependencies; `@cloudflare/vite-plugin` is
  added; `wrangler` stays for `deploy`.
- `wrangler.jsonc`: the `run_worker_first` block (lines 17-20) is deleted —
  `@cloudflare/vite-plugin` emits client assets and the worker into separate
  directories, so worker source cannot leak as a static asset. `main` and
  `assets` are repointed to the plugin's output layout. The `routes`
  custom-domain block and the `observability` block must carry over unchanged.
- `apps/site/src/pages/docs/deployment.mdx` is rewritten: the two-pass build
  section is removed, the preview section updated, the wrangler section updated.

## Testing strategy

- `packages/vite/src/__tests__/hono-preact.test.ts` is heavily rewritten. It
  currently asserts exact plugin-array indices (`plugins[7]`, `plugins[8]`) and
  the `mode`-based two-pass config, all of which is going away. New assertions:
  `adapter` is required; the framework plugin chain is correctly spliced with
  `adapter.vitePlugins()`; core app module and entry wrapper are both generated.
- Per-adapter unit tests: `vitePlugins()` returns the expected plugin set;
  `wrapEntry()` produces the expected tail (Cloudflare re-export shape; Node
  outer-app `serveStatic` + `serve()` shape).
- Two WebSocket dev-server integration tests (Node and Cloudflare), per the
  WebSockets section.
- The `leak-test` fixture (`packages/vite/src/__tests__/fixtures/leak-test/vite.config.ts`)
  is updated to pass an adapter.
- An `apps/example-node` build-and-run smoke test.

## Risks & open questions

- **`@cloudflare/vite-plugin` ↔ Vite 8.0.8 compatibility.** The repo is pinned
  to Vite 8.0.8 (root `package.json` override). This is the hard go/no-go gate
  and should be confirmed by a spike before implementation begins.
- **Worker-source-leak protection.** The current `run_worker_first` list
  guards against the worker bundle being served as a public static asset. The
  migration assumes `@cloudflare/vite-plugin`'s separate output directories
  close that hole; this must be confirmed against the plugin's actual output
  layout.
- **`@preact/preset-vite` under the workerd/SSR environment.** Its JSX/SSR
  transform behavior inside the plugin-managed environments needs a smoke test.
- **Framework-owned Node dev middleware.** D2 means the framework owns a small
  Environment-API dev middleware. Vite's module runner handles HMR, but SSR
  dev-server edge cases (error overlay integration, edge HMR behavior) become
  framework-owned surface and need explicit testing.
- **Client/server discriminator audit.** Gutting `configPlugin` removes the
  `mode`-based discriminator; each framework plugin that branched on it must be
  re-checked (`serverOnlyPlugin` is already `ssr`-based and safe; `client-shim`,
  `client-entry`, `module-key`, and `guard-strip` must each be verified during
  implementation).

## Out of scope / future

- Bun, Deno, Vercel adapters (the interface must not preclude them).
- Moving the generated entry files from disk to virtual modules.
- A framework WebSocket abstraction.
