# hono-preact-ui v0.3.0: release notes

**Released:** 2026-07-17

The library's first breaking release. It reshapes `Select` / `Combobox`
selection into a union discriminated on `multiple`, spells the empty-controlled
state as an explicit `null` rather than an ambiguous `undefined`, and makes a
programmatic clear observable through `onValueChange`. `MenuRadioGroup` gains a
value-type generic. The public-export surface was diffed against the
`hono-preact-ui@0.2.0` tag from a fresh build of both trees: **four exports
added, none removed or renamed**; the breaking part is in the shape of existing
types and the wire behavior of clear, not in the export list.

Ships alongside framework `v0.11.0`, on its own line and tag namespace.

## Breaking changes

### 1. Discriminated `Select` / `Combobox` selection props (#268)

`SelectionProps<Value>` is now a **union discriminated on `multiple`**, and the
empty-controlled state is spelled `null` instead of an ambiguous `undefined`:

- **Single (default):** `value?: Value | null`,
  `onValueChange?: (value: Value | null) => void`.
- **Multiple (`multiple: true`):** `value?: readonly Value[]`,
  `onValueChange?: (value: Value[]) => void`.

`null` means **controlled-and-empty**; `undefined` stays **uncontrolled**. This
removes the old collision where `undefined` had to mean both. Clear is now
**observable**: `Combobox.Clear` and a form reset emit `null` (single) / `[]`
(multiple) through `onValueChange` instead of silently swallowing the change
(the internal casts that hid those emissions are gone).

**Migration:**

- A single-select consumer that modeled "cleared" as `value={undefined}` while
  otherwise controlling `value` should switch to `value={null}`. Leave `value`
  off entirely only if the component is meant to be uncontrolled.
- A single-select `onValueChange` handler now receives `Value | null`; handle
  the clear case.
- A multi-select `onValueChange` now receives `Value[]`, and `value` takes a
  `readonly Value[]`.

### 2. `useListboxSelection` reshaped to `values` / `setValues`

Custom roots built directly on `useListboxSelection` must move from the old
`value` / `setValue` (`Value | Value[]`) to the reshaped
`values: readonly Value[]` / `setValues: (next: Value[]) => void`. The single
vs. multiple distinction now lives in the props layer, not in this hook, so the
hook always works in terms of an array. The new `normalizeSelectionProps` export
maps either arm of `SelectionProps` (single or multiple, controlled or
uncontrolled) onto that internal `readonly Value[]` state, so a custom root can
reuse the same normalization the built-in components do.

## Also in this release (non-breaking)

- **`MenuRadioGroup` carries a value-type generic.** `MenuRadioGroupProps<V extends string = string>`
  narrows `value` / `defaultValue` / `onValueChange` to `V` instead of bare
  `string`. The `= string` default keeps existing untyped usage
  source-compatible; annotate `MenuRadioGroup<'a' | 'b'>` to get the narrowed
  callback.

## New exports

- **Value:** `normalizeSelectionProps` (maps `SelectionProps` onto the internal
  `readonly Value[]` selection state).
- **Types:** `SingleSelectionProps`, `MultipleSelectionProps`,
  `NormalizedSelection`.

No exports were removed or renamed; `SelectionProps`, `useListboxSelection`, and
`ListboxSelection` keep their names while changing shape (see Breaking changes).

## Versioning

`hono-preact-ui` versions independently of the `hono-preact` framework. This
release ships alongside framework `v0.11.0` but on its own line and tag
namespace (`hono-preact-ui@0.3.0`). Still pre-1.0: stable enough to build on,
may evolve ahead of 1.0.

## Install

```bash
npm install hono-preact-ui
```

Requires `preact` (>=10.11.0) as a peer dependency.
