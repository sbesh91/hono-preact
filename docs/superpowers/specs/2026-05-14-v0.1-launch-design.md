# v0.1 Launch — Design

**Date:** 2026-05-14
**Status:** Draft
**Issue:** [#37](https://github.com/sbesh91/hono-preact/issues/37) (item 10 of v0.1 sequencing)
**Parent spec:** `docs/superpowers/specs/2026-05-09-v0.1-framework-direction.md` (section 11, item 10; section 12)

## TL;DR

Convert `apps/app` from an internal demo into a public launch artifact: a single Cloudflare Workers deploy that hosts three top-level URL sections off one origin — `/` (marketing landing), `/docs/*` (the 18 existing MDX pages), `/demo/*` (a feature-showcase mini issue tracker that exercises every v0.1 primitive). Ship `hono-preact@0.1.0` to npm. Tag `v0.1.0`. Cut a GitHub release.

The framework is the site is the proof.

## 1. Goals and non-goals

**Goal.** Ship v0.1 publicly. A first-time visitor at `/` sees the pitch, the four-file code block, and a "Get started" CTA on first paint. `npm install hono-preact` works.

**Success criteria.**
- Deployed site reachable at the URL the user (deploys-handler) provides; rendered by `hono-preact` itself.
- `pnpm publish` succeeds for the umbrella (`hono-preact`) plus the three transitives (`@hono-preact/iso`, `@hono-preact/server`, `@hono-preact/vite`).
- `npm install hono-preact` from a fresh project resolves and works.
- Git tag `v0.1.0`. GitHub release exists with notes.
- All open v0.1 milestone issues closed; milestone itself closed.

**Non-goals (deferred to post-v0.1, tracked).**
- `create-hono-preact` scaffold CLI. Spawn issue at launch.
- Custom domain registration / DNS. The user is handling deploys; `homepage` field gets filled in at publish time.
- Separate demo deploy or subdomain. Single deploy, three URL sections.
- New framework primitives. v0.1 ships what's already in `main`.
- Changelog automation, npm provenance signing, SBOMs, security.txt. All fine to add post-launch.
- Doc search, dark-mode polish, screenshots. Add when traction justifies.

## 2. Site architecture

Single Cloudflare Workers deployment, three top-level URL sections, all built from one app:

```
/                                             marketing landing (Bun-style)
/docs                                         docs index
/docs/<topic>                                 existing 18 MDX pages (unchanged paths)
/demo                                         demo index page
/demo/login                                   mock magic-link login
/demo/projects                                project list
/demo/projects/:projectId                     project layout + issue list
/demo/projects/:projectId/issues/:issueId     issue detail + comments + activity
```

**Mechanics.**
- `apps/app/src/routes.ts` is the single source of truth for all three sections.
- **Rename `apps/app` → `apps/site`.** Reflects what it now is. Update root `package.json` scripts (`pnpm dev`, `pnpm build`, `pnpm deploy`) and any `pnpm --filter app` references accordingly.
- The existing `apps/app/src/pages/movies-*`, `pages/watched.*`, `pages/test.*` (and their `__tests__`) get deleted. Replaced by the issue-tracker demo in `pages/demo/*`.
- `apps/site/src/pages/home.tsx` is rewritten from scratch as the marketing landing (Section 3).

**What does not change.**
- Framework packages (`packages/*`).
- The 18 existing docs MDX files keep their on-disk paths and URL paths.
- `DocsRoute` and `DocsLayout` components.

## 3. Landing page (`/`)

A single-screen Bun-style page in `apps/site/src/pages/home.tsx`. SSR'd HTML + CSS, no client-side state, no animation library.

**Hero band.**
- Product name + version pill: `hono-preact v0.1`.
- Headline (~6 words). Working draft: *"A small full-stack framework."*
- Pitch sentence: *"Hono on the edge, Preact in the browser, manifest driven routes, typed RPC, streaming everywhere."*
- Two CTAs: primary **"Get started"** → `/docs/quick-start`; secondary **"See the demo"** → `/demo`.

**Code block band.**
The four-file day-one snippet from parent spec section 12, syntax-highlighted by the same MDX pipeline the docs use:
- `vite.config.ts` — `defineApp()` one-liner
- `src/routes.ts` — `defineRoutes([...])`
- `src/views/home.tsx` — view component
- `src/Layout.tsx` — `<Head />` + `<ClientScript />`

Real, copy-pasteable code. Stacked vertically or side-by-side, whichever lays out cleanly at the chosen breakpoint.

**Feature cards band.** Four cards, one sentence caption each.
1. **Manifest-driven routes** — *Your routes are a data structure, not a directory tree.*
2. **Typed RPC, end to end** — *Loaders and actions are typed functions; the client gets a typed stub.*
3. **Streaming everywhere** — *Loaders, forms, SSE. Built on `ReadableStream`.*
4. **One package, three subpaths** — *`hono-preact`, `hono-preact/server`, `hono-preact/vite`. Nothing else to install.*

**Footer band.** GitHub link, npm link, license. Small.

**Implementation notes.**
- No loader, no server bundle on the landing route — `view`-only.
- Styles extend the existing `apps/site/src/styles/` setup; no new dep.
- Use the framework's own `<Head>` for `<title>` and meta.

## 4. Demo (`/demo/*`) — mini issue tracker

A small, single-tenant issue tracker structured so every v0.1 framework feature has a natural home. Replaces the deleted `movies` / `watched` / `test` / `live-stats` pages.

**URL surface.** As listed in Section 2.

**Data.** In-memory, per-process, seeded deterministically. Lives in `apps/site/src/demo/data.ts`. No D1, no KV, no Workers bindings to set up. Resets on cold start; this is documented on the `/demo` index page ("This is a demo. Data resets when the Worker restarts.").

**Data shape (sketch).**
- `User { id, email, name }`
- `Project { id, slug, name }`
- `Issue { id, projectId, authorId, title, body, status: 'open' | 'closed', createdAt }`
- `Comment { id, issueId, authorId, body, createdAt }`

**Feature → demo surface map.** Each framework primitive lives somewhere natural so the demo doubles as proof.

| Framework feature | Where it shows up |
|---|---|
| `defineRoutes` + nested layouts | `/demo/projects/:projectId` layout wraps issue list and issue detail; layout state survives navigation |
| Multi-loader routes | Issue detail has `serverLoaders` for issue, comments, and activity in parallel |
| Streaming loaders | Comments stream in (intentionally throttled) under a streaming boundary |
| Actions (`<Form>` + RPC) | Create issue, comment, change status, log out |
| Optimistic UI | Status change + new comment appear optimistic before server confirms |
| Page guards | `/demo/projects/*` requires login; redirects to `/demo/login` |
| Action guards | "Close issue" requires being the issue author; raises `ActionGuardError` (403) |
| Streaming forms | New-comment form streams progressive feedback |
| Auth (Hono signed cookies) | Login sets a signed session cookie; logout clears it; loaders read `ctx.c` |
| `<Head>` | Per-page titles ("INF-12 · Crash on save") |
| `<ViewTransitions>` | Navigation between list and detail animates |
| `useRouteChange` | Closes any open modals on navigation |
| Prefetch | Issue list rows prefetch on hover |

**Auth.** Intentionally mocked. The `/demo/login` form accepts any email; on submit, the server action sets a signed session cookie with that email as the identity. The point is the cookie/guard/loader plumbing, not an auth provider integration.

**Styles.** No bespoke design system. Extend `apps/site/src/styles/`. The whole demo should look like part of the site, not a different product.

## 5. Docs (`/docs/*`)

The 18 existing MDX pages stay where they are. Two changes:

### 5.1 `/docs` index refresh

Rewrite `pages/docs/index.mdx` into a navigable docs landing: short orientation paragraph plus a categorized link list.

Categories:
- **Get started** — `quick-start`, `structure`
- **Routing & layouts** — `routes`, `pages`, `layouts`, `vite-config`
- **Data** — `loaders`, `actions`, `optimistic-ui`, `loading-states`, `streaming`
- **Auth & access** — `guards`, `action-guards`
- **Operations** — `prefetch`, `reloading`, `render-page`, `deployment`

Each link is a one-line description of what's on the other side. Navigation surface, not a tutorial.

### 5.2 Content audit

A single pass through all 18 pages, checking four things per page:

1. **Imports.** Every code snippet imports from the current published surface (`hono-preact`, `hono-preact/server`, `hono-preact/vite`). Catch leftover `@hono-preact/*` or pre-#40 paths.
2. **Symbols.** Every API referenced still exists. Specifically catch references to: `createGuard`, `serverGuards`/`clientGuards` arrays, `cacheRegistry`, `useLoaderData<typeof loader>()`, `defineApp` (pre-zero-arg), and any other symbol the sequencing deleted.
3. **Cross-links.** Every internal `/docs/<other>` link resolves.
4. **Demo references.** Any link to `/movies`, `/watched`, `/test` gets rewritten to a `/demo/...` URL on the issue tracker (or deleted if no live counterpart exists).

**Output.** A checklist in the implementation plan, one entry per page, each finding marked either **"fix inline"** (one-line edits, do now) or **"open follow-up issue"** (anything needing >30 min of rewrite). Launch is blocked on no broken imports and no dead links; it is not blocked on perfect docs.

**Pages flagged as likely-stale, audit first:**
- `quick-start.mdx` — most-visited page; highest cost if its snippets don't work.
- `structure.mdx` — describes the on-disk layout; multi-loader-routes and consolidation likely changed this.

### 5.3 Cross-link sweep

Add:
- Marketing landing CTA → `/docs/quick-start`.
- `/demo` index page → the doc page each demo screen exercises (issue detail page footer links to `/docs/loaders` + `/docs/streaming`, etc.).
- `actions.mdx` → SameSite-cookies note (Section 7.1).

### 5.4 Out of scope for launch

Full content rewrite of every doc, screenshots, dark-mode polish, doc search. Tracked as v0.2 candidates if anything specific comes out of the audit.

### 6.7 Pre-publish checks

Run locally before tagging, also wired into CI on a release branch if practical:
- `pnpm build` clean from a fresh `pnpm install`.
- `pnpm test` green.
- `pnpm pack --dry-run` per publishable package; eyeball the file lists for stowaways (no `__tests__/`, no `.tsbuildinfo`, no source maps unless intended).

## 7. Final pieces

### 7.1 SameSite-cookies note

Lives in `/docs/actions.mdx` as a "Security" section near the bottom:

> `<Form>` posts go to `/__actions` on the same origin. The framework relies on `SameSite=Lax` (Hono's cookie default) for CSRF protection. Cross-origin POSTs without a credential do not carry the session cookie; cross-origin POSTs from a malicious site cannot read the response. If you need a stricter posture (explicit-origin CSRF middleware on `/__actions`), that is not user-configurable in v0.1 — see [#43](https://github.com/sbesh91/hono-preact/issues/43) for the post-v0.1 design.

The top-level README's Status section links to this anchor. Not in the marketing landing — too detailed for `/`.

### 7.2 Release process

1. Pre-flight (Section 6.7).
2. Bump versions: all four publishable packages from `0.0.1` → `0.1.0`. One commit: `chore: v0.1.0`.
3. Tag: `git tag v0.1.0 -m "v0.1.0"`. Push with `--tags`.
4. Publish: `pnpm -r publish --access public` from clean main.
5. GitHub release: `gh release create v0.1.0` with notes assembled from:
   - The pitch sentence as the headline.
   - "What's in v0.1" — one bullet per sequenced item from parent spec section 11, paraphrased into user-facing language ("manifest-driven routes," "streaming loaders and forms," "typed Hono context on every seam," etc.).
   - "Known limitations" — links the spawn issues (CSRF reach #43, scaffold post-v0.1, streaming-loader cookie rotation, etc.).
6. Close `v0.1` milestone after #37 closes.

## 8. Risks and open questions

- **Audit balloons.** Mitigation: hard 30-min-per-fix budget; anything over becomes a follow-up issue.
- **`apps/app` → `apps/site` rename is invasive.** Mitigation: do it as the first commit on the launch branch so subsequent commits read against the new name.
- **Demo features that don't exist yet (streaming forms with progressive feedback, `<ViewTransitions>` on navigation).** Need to confirm during plan-writing that every row in the Section 4 feature map maps to a shipped primitive, not an aspirational one. If a row turns out to be aspirational, drop it from the demo or scope a small fix-up.
- **Deploy URL is a runtime dependency for `homepage` and README.** The user is handling deploys; the implementation plan should treat the URL as a single placeholder filled in at the publish step.
