# Loading States

Show a loading UI during client-side navigation by branching on the `loading` argument inside `loader.View()`. While a fetch is in flight, `loading` is `true` and `data` may be `undefined` (cold load) or the previous value (reload). Render your own loading affordance for each state; the framework supplies the state, not the UI.

## Basic usage

```tsx
// src/pages/movies.tsx
import { definePage } from 'hono-preact';
import { serverLoaders } from './movies.server.js';

const moviesLoader = serverLoaders.default;

const MoviesView = moviesLoader.View(({ data, loading }) => {
  if (loading) return <p>Loading...</p>;
  return (
    <ul>
      {data!.movies.results.map((m) => (
        <li key={m.id}>{m.title}</li>
      ))}
    </ul>
  );
});

export default definePage(MoviesView);
```

```ts
// src/routes.ts
{
  path: '/movies',
  view: () => import('./pages/movies.js'),
  server: () => import('./pages/movies.server.js'),
}
```

`loading` is `true` only on client-side navigation; SSR and first-load hydration render with data already present, so `loading` is `false` on the initial paint.

## When loading is true

| Navigation                   | `loading` on entry             |
| ---------------------------- | ------------------------------ |
| SSR (first load)             | `false`. Data is preloaded into the HTML.  |
| Hydration                    | `false`. Client reads from `data-loader` attribute. |
| Client-side nav (cache miss) | `true`, until the loader resolves. |
| Client-side nav (cache hit)  | `false`. Cached data renders immediately. |
| Reload (stale-while-revalidate) | `true`, but `data` retains the previous value. |

## Using a skeleton

Branch on `loading` and return a skeleton component:

```tsx
const MoviesSkeleton = () => (
  <ul>
    {Array.from({ length: 5 }).map((_, i) => (
      <li key={i} class="h-6 w-48 animate-pulse bg-gray-200 rounded" />
    ))}
  </ul>
);

const MoviesView = moviesLoader.View(({ data, loading }) => {
  if (loading) return <MoviesSkeleton />;
  return (
    <ul>
      {data!.movies.results.map((m) => (
        <li key={m.id}>{m.title}</li>
      ))}
    </ul>
  );
});
```

## Stale-while-revalidate

During a reload, `loading` is `true` but `data` holds the previous value. This lets you keep content visible while refreshing:

```tsx
const MoviesView = moviesLoader.View(({ data, loading }) => (
  <div>
    {loading && <span class="text-sm text-muted">Refreshing...</span>}
    <ul>
      {(data?.movies.results ?? []).map((m) => (
        <li key={m.id}>{m.title}</li>
      ))}
    </ul>
  </div>
));
```

## Error handling inside the render function

The `error` argument is non-null when the loader rejects (either on the initial load or on a re-fetch). Use `reload` to let the user retry:

```tsx
const MoviesView = moviesLoader.View(({ data, loading, error, reload }) => {
  if (loading) return <p>Loading...</p>;
  if (error) return (
    <div role="alert">
      <p>Couldn't load movies: {error.message}</p>
      <button onClick={reload}>Retry</button>
    </div>
  );
  return (
    <ul>
      {data!.movies.results.map((m) => (
        <li key={m.id}>{m.title}</li>
      ))}
    </ul>
  );
});
```

## `.View()` render args reference

The render function receives these args on every render:

| Arg      | Type                | Description                                                                                    |
| -------- | ------------------- | ---------------------------------------------------------------------------------------------- |
| `data`   | `Serialize<T> \| undefined` | The loader's last resolved value. `undefined` on a cold load before the first response. Retained from the previous fetch during a reload. |
| `loading` | `boolean`          | `true` while a fetch is in flight (cold load or reload). |
| `error`  | `Error \| null`     | Non-null when the loader errored. Check this alongside `loading` to show a retry affordance. |
| `reload` | `() => void`        | Re-runs the loader immediately. |

Plus any props declared by the generic `P` (see [prop passthrough in loaders](/docs/loaders)).

## Page-level error fallback

`definePage` accepts a page-level `errorFallback` that catches errors from the rest of the page tree (for example, a render-time throw outside any loader boundary):

```tsx
export default definePage(MoviesView, {
  errorFallback: (err) => <p>Something went wrong: {err.message}</p>,
});
```

Loader-specific loading and error UI lives inside the `.View()` render function. The page-level `errorFallback` is the outer safety net.
