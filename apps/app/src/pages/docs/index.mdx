# hono-preact

Fullstack made easy.

Routing is one file. Pages are file pairs: a `.tsx` view and a sibling `.server.ts` that never reaches the browser. The framework handles SSR, data fetching, mutations, and access control. You write the logic.

```ts
// routes.ts: every URL in the app, in one place
import { defineRoutes } from '@hono-preact/iso';

export default defineRoutes([
  { path: '/', view: () => import('./pages/home.js') },
  {
    path: '/movies',
    view: () => import('./pages/movies.js'),
    server: () => import('./pages/movies.server.js'),
  },
]);
```

```ts
// movies.server.ts: runs only on the server
const serverLoader = async () => ({ movies: await getMovies() });
export default serverLoader;
export const loader = defineLoader(serverLoader);
```

```tsx
// movies.tsx: the view component, with bindings attached via definePage
import { definePage } from '@hono-preact/iso';
import { loader } from './movies.server.js';

function Movies() {
  const { movies } = loader.useData();
  return <ul>{movies.map((m) => <li key={m.id}>{m.title}</li>)}</ul>;
}
export default definePage(Movies, { loader });
```

On the first request, the server runs the loader directly and preloads the result into the HTML. On hydration, the client reads from that preloaded data. On client-side navigation, the framework calls the loader over RPC. Same loader, three environments, zero config.

**[→ Quick Start](/docs/quick-start)**: build a page with data and a mutation in about 10 minutes.

## What you author

| File | Purpose |
|---|---|
| `src/routes.ts` | The route table. One entry per URL. Declares which view (and optionally which `.server.ts`) lives at each path. |
| `src/pages/<name>.tsx` | View component. Default-exports the page; uses `definePage` when it has loader or guard bindings. |
| `src/pages/<name>.server.ts` | Server-only sibling. Holds the default-exported loader plus optional `loader`, `serverActions`, `serverGuards`, `actionGuards` named exports. |
| `src/Layout.tsx` | The HTML shell rendered around every page. Default-exports a component. |
| `src/api.ts` (optional) | A Hono app default-exporting any custom REST routes you need. The framework mounts it before its own SSR catch-all. |

## The server/client boundary

Two Vite plugins enforce the boundary between server and client code. During the client bundle build, `serverOnlyPlugin` rewrites `*.server.*` imports: the default export becomes an RPC function, the `loader` named export becomes a client-safe `LoaderRef` (carrying its own auto-attached cache) that calls the same RPC, any imported `serverGuards` or `actionGuards` become empty arrays, and `serverActions` becomes a Proxy. Side-effect-only `.server.*` imports are stripped entirely. Dynamic `() => import('./*.server.*')` calls (used by the `server` field in `routes.ts`) are rewritten to `Promise.resolve({})` on the client. Database clients, secrets, and internal API URLs in `.server.*` files never reach the browser bundle.

`serverLoaderValidationPlugin` enforces the contract at build time: `.server.*` files may only export `loader`, `serverGuards`, `serverActions`, or `actionGuards` as named exports (alongside the default loader). Any other named export fails the build, and re-exporting from a `.server.*` module (`export { loader } from './x.server.js'`) fails the build too.
